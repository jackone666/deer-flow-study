# 12 端到端深挖：把 Agent 平台讲成一个完整项目

这一篇解决一个面试里的核心问题：

> 前面每个模块都能单独讲，但面试官真正想听的是：你到底做了一个什么系统？一次请求进来以后，系统怎么跑？哪里体现工程复杂度？你怎么证明这个设计不是堆概念？

所以本篇不再按单点模块展开，而是按“项目故事 + 运行链路 + 面试追问”组织。目标是让你能把 `interview-agent-platform` 从零散知识点讲成一个完整的 Agent Harness 项目。

## 一句话定位

我做的是一套面向复杂任务的 Agent Harness 平台。它不是只把 LLM 包成聊天接口，而是把一次 Agent 任务拆成可观测、可治理、可安全执行的运行时流程：入口层负责会话和任务生命周期，状态层负责合并消息和产物，上下文层负责动态注入记忆和系统提醒，工具层负责权限、检索和延迟加载，安全层负责 Guardrails 和 Sandbox，任务结束后再通过 Memory、Skill 和 Eval 形成持续改进闭环。

## 面试官最关心的不是“用了什么”

很多候选人会这样讲 Agent 项目：

> 我用了 LangGraph / MCP / 向量库 / Sandbox / Memory。

这不够。面试官会继续追问：

- 这些模块为什么要拆开？
- 每一层的输入输出是什么？
- 哪些逻辑同步执行，哪些逻辑异步执行？
- 工具多了以后怎么控制 schema 预算？
- 上下文长了以后怎么压缩又不丢关键事实？
- Agent 调危险工具时谁拦？
- 某一次运行失败后怎么定位是模型、工具、上下文还是沙箱问题？

所以更好的讲法是：

> 我不是围绕某个框架做封装，而是围绕 Agent 运行时的不确定性做分层治理。模型负责推理，但平台负责把输入、上下文、工具、安全、状态和观测变成稳定的工程边界。

## 项目可以拆成七层

| 层级 | 解决的问题 | 关键能力 | 面试价值 |
| --- | --- | --- | --- |
| Gateway / API 层 | 用户请求如何进入系统 | 线程、运行、鉴权、上传、流式响应 | 说明不是 demo，而是服务化系统 |
| Runtime 层 | 一次任务如何被启动、恢复、取消 | run manager、worker、thread_id、工作目录 | 说明你理解长任务生命周期 |
| State 层 | 多轮消息和工具产物如何合并 | ThreadState、reducer、artifact、promoted tools | 说明你理解状态一致性 |
| Context 层 | 模型调用前应该看到什么 | 动态上下文、长期记忆、日期、系统提醒 | 说明你理解上下文工程 |
| Tooling 层 | 大量工具如何治理 | 分组、allowed-tools、deferred tools、tool_search | 说明你理解工具规模化 |
| Safety 层 | 工具执行前如何兜底 | Guardrails、Sandbox、审计、fail-closed | 说明你有安全边界意识 |
| Learning 层 | 系统如何持续变好 | Memory、Skill、Eval、Trace、数据飞轮 | 说明你有闭环思维 |

面试时不要一口气把表背完。可以先说：

> 我把它拆成运行时、状态、上下文、工具、安全和学习闭环六个核心层。每一层都有明确边界，模型只负责下一步决策，平台负责让决策可控、可追踪、可恢复。

## 一次 Agent 运行的完整链路

下面这条链路最适合作为 10 分钟项目讲解主线。

```text
用户请求
  -> Gateway 创建或加载 thread
  -> Runtime 创建 run，绑定 thread_id / user_id / agent_id
  -> 读取历史消息、附件、产物、线程元数据
  -> 进入 Agent Harness
  -> Context middleware 注入动态上下文
  -> Summarization middleware 判断是否压缩历史
  -> Tooling middleware 装配可用工具
  -> 模型选择下一步：回答 / 调工具 / 派子任务
  -> 工具调用前经过 Guardrails
  -> 文件、命令、代码执行进入 Sandbox
  -> 工具结果写回 ThreadState
  -> 流式事件返回前端
  -> 任务结束后异步触发 Memory / Skill / Eval
  -> Trace、Metrics、日志进入观测和数据飞轮
```

这条链路背后的设计重点是：主路径要尽可能稳定，非关键的学习和沉淀放到异步链路；高风险动作必须同步拦截；高成本上下文和工具 schema 必须按需进入模型窗口。

## 为什么叫 Harness，不叫 Chatbot

普通 Chatbot 的核心是：

```text
messages -> model -> response
```

Agent Harness 的核心是：

```text
state + context + tools + policies + runtime -> model decision -> controlled execution -> state update
```

区别在四点：

1. **状态不是纯聊天记录**：还包括产物、图片、工具提升状态、线程元信息、运行事件。
2. **上下文不是全部历史**：要区分 system、dynamic context、memory、summary、recent messages。
3. **工具不是全量暴露**：要按权限、任务、技能和检索结果动态装配。
4. **模型不能直接执行世界动作**：所有工具调用都经过权限、安全和沙箱边界。

可以这样回答：

> Chatbot 重点是生成回复，Harness 重点是管理一次复杂任务的生命周期。它要决定模型看到什么、能用什么工具、每一步怎么落状态、失败怎么恢复、危险动作怎么拦截，以及任务结束后哪些经验可以沉淀。

## 核心对象怎么讲

如果面试官让你说“系统里有哪些核心对象”，可以按下面讲。

| 对象 | 作用 | 不应该承载什么 |
| --- | --- | --- |
| `Thread` | 表示一段持续会话，承载历史消息和元数据 | 不直接等同于一次运行 |
| `Run` | 表示一次执行任务，有开始、取消、完成、失败状态 | 不保存所有业务逻辑 |
| `ThreadState` | Agent 图内的工作状态，合并消息、产物、工具状态 | 不负责权限判断 |
| `Middleware` | 在模型调用或工具调用前后插入横切逻辑 | 不应该互相隐式依赖 |
| `Tool` | 模型可调用的动作接口 | 不应该绕过权限和沙箱 |
| `Sandbox` | 受控执行环境 | 不负责判断业务意图是否安全 |
| `Memory` | 用户偏好、事实、纠偏的长期沉淀 | 不保存完整聊天日志 |
| `Skill` | 可复用工作流和操作经验 | 不保存用户个人隐私 |
| `Trace` | 一次运行的可观测链路 | 不作为业务状态源 |

面试表达要点：

> 我尽量让对象职责单一。Thread 管会话，Run 管一次执行，State 管图内数据，Middleware 管横切策略，Tool 管能力入口，Sandbox 管执行边界，Memory 和 Skill 管任务后的沉淀。

## 关键数据流 1：上下文注入

动态上下文注入发生在模型调用前。它的输入通常包括：

- 当前用户和 Agent 标识。
- 当前线程历史。
- 长期记忆摘要。
- 当前日期和环境提醒。
- 最近是否已经注入过相同提醒。

输出是模型调用前的消息列表增强，而不是直接改写用户消息。

典型处理：

```text
读取 ThreadState.messages
  -> 判断是否需要注入动态上下文
  -> 拉取用户长期记忆和当前日期
  -> 生成隐藏上下文消息
  -> 插入到模型可见消息区
  -> 标记 injected，避免重复注入
```

这里最容易被追问的是：为什么不把所有记忆放 system prompt？

可以答：

> system prompt 更适合放稳定规则，而动态记忆和日期是运行时信息。把动态信息和稳定规则混在一起会导致优先级、可追踪性和更新边界不清楚。单独作为动态上下文注入，可以记录来源、控制 token 预算，也方便摘要中间件保护它。

## 关键数据流 2：摘要压缩

摘要压缩不是“历史太长就总结一下”这么简单。它要解决三件事：

1. 哪些消息可以压缩。
2. 哪些消息必须保留。
3. 压缩后如何保持工具调用链路和系统提醒不坏。

典型处理：

```text
计算当前消息 token
  -> 超过阈值才触发
  -> 找到 cutoff
  -> 保留最近消息、动态提醒、关键 tool 对齐消息
  -> 将旧消息生成结构化 summary
  -> 用 summary message 替换旧历史
  -> 继续进入模型调用
```

面试重点：

> 摘要压缩的目标不是尽可能短，而是在 token 预算内保留任务状态。比如用户目标、已做决策、失败尝试、当前待办、工具产物路径都比礼貌寒暄更重要。

## 关键数据流 3：工具治理

工具治理的本质是控制模型的行动空间。

如果把所有工具直接塞给模型，会有三个问题：

- schema 太多，占上下文。
- 相似工具太多，模型容易选错。
- 权限边界模糊，某个技能不该拥有的工具也被看见。

更好的链路是：

```text
加载全部工具 catalog
  -> 根据 agent / skill / config 做第一层过滤
  -> 常用工具直接绑定
  -> 大量长尾工具进入 deferred catalog
  -> 模型需要时调用 tool_search
  -> 检索命中后提升工具
  -> 后续调用只暴露被提升的工具 schema
```

可以这样讲 trade-off：

> 全量绑定最简单，但 schema 成本和误调用风险高。延迟加载多了一次检索成本，但能把模型上下文留给任务本身，也能让权限控制更清晰。

## 关键数据流 4：Guardrails 和 Sandbox

很多人会把 Guardrails 和 Sandbox 混在一起。面试时一定要讲清楚：

- Guardrails 判断“这件事应不应该做”。
- Sandbox 控制“即使要做，也只能在受限环境里做”。

例如模型想执行命令：

```text
模型生成 tool call
  -> Tool middleware 捕获调用
  -> Guardrails 根据工具名、参数、上下文做 allow / deny
  -> deny：返回标准化 ToolMessage，让模型知道被拒绝
  -> allow：进入 Sandbox 执行
  -> Sandbox 限制目录、网络、资源、环境变量和审计
  -> 执行结果写回状态
```

面试里的高分表达：

> 我不会只靠 prompt 提醒模型不要做危险事。Prompt 是软约束，Guardrails 是策略判断，Sandbox 是执行隔离。即使模型被 prompt injection 诱导，危险动作也要在工具执行前被同步拦截。

## 关键数据流 5：任务结束后的学习闭环

主任务结束不代表系统结束。Agent 平台要把经验沉淀成三类资产：

| 资产 | 来源 | 用途 |
| --- | --- | --- |
| Memory | 用户明确偏好、纠错、稳定事实 | 下一轮个性化和少重复解释 |
| Skill | 多次成功的工具流程、操作步骤、踩坑经验 | 下次同类任务直接复用 |
| Eval Case | 失败样本、边界样本、代表性任务 | 回归测试和质量对比 |

推荐讲法：

> 我把学习闭环放在异步链路，不阻塞主回答。Memory 记录用户长期偏好，Skill 记录可复用流程，Eval case 记录质量验证样本。三者都要经过门禁，不能把一次偶然成功或包含隐私的数据直接写回系统。

## 难点一：状态一致性

Agent 运行中会不断产生消息、工具结果、产物、图片、子任务返回、提升后的工具。不能用简单 dict 覆盖状态，因为并发或分支执行时容易丢字段。

更合理的是 reducer 思路：

```text
old_state + partial_update -> merged_state
```

不同字段有不同合并策略：

| 字段 | 合并策略 |
| --- | --- |
| messages | 追加或按 message id 合并 |
| artifacts | 按路径或 id 更新 |
| promoted_tools | 去重合并 |
| token_usage | 累加 |
| errors | 追加并保留来源 |
| run_status | 按状态机迁移 |

面试回答：

> ThreadState 不是普通 dict，而是运行时协议。不同字段的合并语义不同，所以需要 reducer 明确规定如何追加、去重、覆盖和累加，避免工具结果或子任务结果互相覆盖。

## 难点二：同步和异步边界

一个稳定 Agent 平台必须明确哪些事同步做、哪些事异步做。

| 同步 | 原因 |
| --- | --- |
| 鉴权和用户隔离 | 不通过不能进入主流程 |
| 动态上下文注入 | 影响当次模型决策 |
| 工具权限过滤 | 决定模型能看到什么 |
| Guardrails | 高风险动作必须立即拦截 |
| Sandbox 执行 | 工具结果影响下一步推理 |

| 异步 | 原因 |
| --- | --- |
| 长期记忆更新 | 不应阻塞当前回复 |
| Skill 沉淀 | 需要质量门禁和人工检查 |
| Eval 回放 | 成本较高，适合后台跑 |
| 指标聚合 | 不影响本次任务语义 |
| 日志清洗 | 只服务后续分析 |

高分表达：

> 同步链路只放影响当次正确性和安全性的逻辑；异步链路放学习、统计和沉淀。这样既保证安全，又不让主路径被后台分析拖慢。

## 难点三：用户隔离

Agent 平台做长期记忆、上传文件和工具执行时，用户隔离非常关键。

需要隔离的对象包括：

- thread 和 run。
- 上传文件和生成产物。
- 长期记忆。
- sandbox 工作目录。
- tool credential。
- trace 和日志中的敏感字段。

可讲设计：

```text
user_id + thread_id + agent_id
  -> 作为读取 memory 的 scope
  -> 作为 run 查询和取消的 scope
  -> 作为 sandbox 工作目录和 artifact 路径的一部分
  -> 作为 trace 查询权限边界
```

如果面试官追问“为什么 thread_id 不够”，可以答：

> thread_id 标识会话，但权限边界应该由 user_id 决定。只用 thread_id 容易在接口越权或 ID 泄露时访问到别人的运行和记忆。所有读取都应该同时校验 user scope。

## 难点四：可观测性

Agent 出问题时，不能只看最终回答。要能沿着 trace 找到失败位置。

一次运行至少要记录：

| 维度 | 字段 |
| --- | --- |
| 身份 | run_id、thread_id、user_id、agent_id |
| 模型 | model、input_tokens、output_tokens、latency |
| 上下文 | message_count、summary_triggered、memory_injected |
| 工具 | tool_name、args_hash、status、duration、error_type |
| 安全 | guardrail_decision、policy_name、sandbox_id |
| 质量 | task_success、user_retry、eval_score、human_label |

排查链路：

```text
先看 run 是否完成
  -> 再看模型是否超时或 token 爆了
  -> 再看 summary 是否触发并丢了关键上下文
  -> 再看 tool_search 是否召回错误工具
  -> 再看 Guardrails 是否误杀
  -> 再看 Sandbox 是否资源不足或网络失败
```

面试表达：

> 我会按层定位，而不是直接说模型不行。Agent 失败可能来自上下文、工具、权限、安全策略、沙箱、网络、模型本身或用户输入。Trace 的价值就是把一次复杂运行拆成可定位的 span。

## 可以准备的项目亮点

### 亮点 1：上下文不是越多越好

我做了动态注入和摘要压缩，把长期记忆、当前日期、系统提醒、最近消息和摘要区分开。这样模型看到的是当次任务需要的信息，而不是完整历史堆叠。

### 亮点 2：工具不是越多越好

工具层做了分组、权限过滤和延迟加载。常用工具直接绑定，长尾工具通过 tool_search 检索后提升，减少 schema 预算和误调用。

### 亮点 3：安全不是一句 prompt

工具调用前有 Guardrails，执行时有 Sandbox，拒绝时返回标准化 ToolMessage，provider 异常时 fail-closed。安全策略在执行链路上，而不是只写在提示词里。

### 亮点 4：学习闭环有门禁

Memory、Skill、Eval 都不是无脑写入。用户纠偏、稳定事实、可复用流程和失败样本要分层沉淀，并通过格式、来源、效果和隐私门禁。

### 亮点 5：可观测性能解释失败

每次运行记录模型、上下文、工具、安全、沙箱和质量信号。遇到慢、错、失败、误杀，都能按 span 排查。

## 10 分钟项目讲稿

可以按下面这个版本练习，不用逐字背。

> 我做的是一个面向复杂任务的 Agent Harness 平台。它不是简单聊天机器人，而是把一次 Agent 任务拆成运行时、状态、上下文、工具、安全和学习闭环几层。
>
> 一次请求进来后，Gateway 会创建或加载 thread，Runtime 创建 run，并绑定 user_id、thread_id 和 agent_id。进入 Harness 后，系统先读取历史消息、附件和线程元数据，然后在模型调用前通过中间件注入动态上下文，比如长期记忆、当前日期和系统提醒。如果历史消息超过 token 阈值，摘要中间件会压缩旧消息，但保留最近消息、动态提醒和关键工具调用链路。
>
> 工具侧我没有把所有工具都直接暴露给模型，而是先按配置、技能权限和分组过滤。常用工具直接绑定，大量长尾工具进入 deferred catalog，模型需要时通过 tool_search 检索并提升，这样能减少 schema 预算，也降低误调用概率。
>
> 当模型决定调用工具时，工具调用不会直接执行，而是先经过 Guardrails。Guardrails 根据工具名、参数和上下文做 allow / deny 判断；如果拒绝，会返回标准化 ToolMessage，让模型能继续规划；如果允许，文件、命令或代码执行会进入 Sandbox。Sandbox 负责隔离工作目录、资源和环境变量，并记录审计信息。
>
> 运行过程中所有消息、工具结果、产物和提升后的工具都会写回 ThreadState。ThreadState 不是简单 dict，而是通过 reducer 合并不同类型更新，避免并发或子任务返回时互相覆盖。
>
> 任务结束后，我把不影响当前回答的学习逻辑放到异步链路：用户偏好和纠错进入 Memory，多次成功流程沉淀成 Skill，失败样本和边界样本进入 Eval case。再通过 trace、metrics 和 dashboard 观察质量、成本、安全和工具命中率，形成数据飞轮。
>
> 这个项目最核心的价值是把 Agent 的不确定性工程化：模型负责推理，平台负责上下文预算、工具治理、安全边界、状态一致性和持续评估。

## 20 分钟深入讲法

如果面试官愿意听更细，可以按五个深挖点展开。

### 1. 为什么做动态上下文

先讲问题：长对话里模型不能每轮都看完整历史，长期偏好也不应该散落在聊天记录里。

再讲设计：在模型调用前注入长期记忆、日期、系统提醒，作为隐藏上下文进入模型窗口。

最后讲权衡：动态注入提升相关性，但要控制重复注入、token 预算和 prompt injection 风险。

### 2. 为什么要摘要压缩

先讲问题：上下文窗口有限，工具调用多时 token 增长很快。

再讲设计：超过阈值后压缩旧消息，保留最近消息、动态提醒和关键 tool 对齐关系。

最后讲权衡：摘要会损失细节，所以需要摘要 rubric、回归样本和关键字段保护。

### 3. 为什么工具要治理

先讲问题：工具多会导致 schema 爆炸、误调用、权限混乱。

再讲设计：工具 catalog、分组、allowed-tools、deferred tools、tool_search、promoted state。

最后讲权衡：延迟加载多一次检索，但换来更低上下文成本和更强权限边界。

### 4. 为什么安全要分层

先讲问题：Agent 可以执行文件、命令、网络请求，风险比普通问答高。

再讲设计：Guardrails 判断动作意图，Sandbox 限制执行环境，审计记录实际行为。

最后讲权衡：fail-closed 可能误杀，但对高风险执行更可靠；误杀可以通过策略和白名单优化。

### 5. 为什么要评估闭环

先讲问题：Agent 的好坏不能只靠主观感觉。

再讲设计：离线 eval、线上 trace、用户反馈、失败样本、数据门禁和回归集。

最后讲权衡：评估会增加成本，但能防止优化只在个别样本上变好。

## 常见追问与高质量回答

### Q1：你这个项目最难的点是什么？

不要只说“上下文很难”。可以说：

> 最难的是把模型的不确定性收敛成工程边界。上下文会膨胀，工具会误选，记忆会过期，安全策略会误杀，沙箱会失败。我的做法不是指望模型一次性变聪明，而是把每类不确定性放到对应层治理：Context 管输入，Tooling 管行动空间，Guardrails 管策略，Sandbox 管执行边界，Trace 管定位和评估。

### Q2：你怎么证明这个平台不是 prompt engineering？

> Prompt 只是其中一层。真正的平台能力体现在状态管理、工具治理、安全执行和可观测性。比如工具权限不是靠 prompt 说“不要用”，而是在装配工具时就过滤；危险动作不是靠 prompt 提醒，而是在 tool call 前被 Guardrails 拦截；文件命令不是直接在宿主机执行，而是在 Sandbox 里执行。

### Q3：如果模型选错工具怎么办？

> 先减少选错概率，再处理选错结果。减少概率靠工具描述规范、分组、allowed-tools 和 tool_search 召回评估；处理结果靠工具错误标准化、可重试策略和 trace。对于高风险工具，即使模型选中了，也要经过 Guardrails 和 Sandbox。

### Q4：如果长期记忆错了怎么办？

> 记忆要支持纠偏和冲突删除。用户明确说“不是这样”时，系统要把它识别为 correction，而不是追加一条相互矛盾的新事实。记忆还要保留来源和更新时间，越新的用户明确纠偏优先级更高。

### Q5：为什么任务结束后才更新记忆？

> 主回答路径要低延迟、可控。长期记忆更新需要抽取事实、判断冲突、过滤隐私和落库，如果同步做会拖慢用户体验。异步更新牺牲的是下一轮立刻可见的一致性，但换来主路径稳定。对强一致需求可以在当前 ThreadState 里保留短期事实。

### Q6：如果 Guardrails provider 挂了怎么办？

> 对危险工具应该 fail-closed，返回标准化拒绝消息，而不是默认放行。这样可用性会受影响，但安全边界不会被 provider 故障击穿。低风险工具可以按策略降级，但命令执行、文件写入、网络访问这类动作不能无保护执行。

### Q7：如果 Sandbox 很慢怎么办？

> 先通过 trace 区分是启动慢、命令慢、网络慢还是资源不足。优化方向包括线程级复用、懒加载、预热常用镜像、限制并发、设置超时、缓存依赖和回收孤儿实例。但我不会用本地裸执行作为默认 fallback，因为那会破坏安全边界。

### Q8：如果摘要丢了关键细节怎么办？

> 摘要要有结构化 rubric，明确必须保留用户目标、约束、已完成步骤、失败尝试、工具产物路径、未完成 todo 和关键决策。其次要保留最近消息，不把所有历史都压成摘要。最后用回归集评估摘要前后任务成功率。

### Q9：如何处理子 Agent？

> 子 Agent 用来隔离复杂子任务的上下文和工具空间。主 Agent 负责拆解任务和整合结果，子 Agent 负责某个明确目标，比如检索、代码检查或文档生成。子任务返回的是结构化结果，不应该把子 Agent 的完整上下文全部塞回主上下文。

### Q10：这个系统如果重新做，你会优化什么？

> 我会优先优化三块：第一是更细粒度的 eval case 管理，把失败样本自动分类到上下文、工具、安全、沙箱几类；第二是更强的工具语义检索，从 TF-IDF 升级到 embedding + rerank；第三是更完整的安全策略版本化，让每次 Guardrails 决策能关联到 policy version，方便回滚和 A/B。

## 项目复盘 STAR 版本

面试官如果问“讲一个你做过的项目”，可以按 STAR 组织。

### Situation

复杂任务 Agent 在长对话、多工具、多文件场景下容易失控：上下文越来越长，工具 schema 占满窗口，模型会误调用工具，文件和命令执行存在安全风险，失败后也很难定位原因。

### Task

我要设计一套 Agent Harness，让模型在复杂任务中仍然可控、可追踪、可扩展，并支持长期记忆和经验沉淀。

### Action

我把系统拆成运行时、状态、上下文、工具、安全和学习闭环。运行时管理 run 生命周期；ThreadState 用 reducer 合并消息、产物和工具状态；动态上下文和摘要压缩控制模型输入；工具层用 allowed-tools、分组和 deferred tools 控制行动空间；Guardrails 和 Sandbox 控制危险动作；任务结束后异步更新 Memory、Skill 和 Eval case，并用 Trace 和 Metrics 做观测。

### Result

这个设计让 Agent 从单轮聊天变成可治理的任务执行系统。它能处理长对话、工具膨胀、安全执行和失败定位问题，也方便后续扩展新工具、新安全策略、新评估集和新的 Agent 能力。

如果面试快结束，可以这样收束：

> 这个项目我最想强调的是，我把 Agent 当成一个工程系统来做，而不是只调 prompt。核心设计是用 Harness 管理运行时，用 Context 管理模型输入，用 Tooling 管理行动空间，用 Guardrails 和 Sandbox 管理安全边界，用 Memory、Skill 和 Eval 管理持续改进。这样系统既能完成复杂任务，也能解释为什么成功、为什么失败，以及下一步怎么优化。
