# Agent 平台项目面试准备总览

这份文档用于准备“智能 Agent 平台”项目的技术面试。重点不是背概念，而是把项目讲成一个完整工程系统：为什么做、怎么设计、核心链路是什么、遇到什么问题、如何权衡。

## 项目定位

一句话：

> 我设计并实现了一套面向复杂任务的智能 Agent 平台，核心解决长上下文、多轮记忆、多工具治理和安全执行问题，让 Agent 在长对话、多工具、多任务场景下仍然可控、可追踪、可扩展。

面试官会重点看五类能力：

1. **上下文工程能力**：是否理解模型上下文不是越多越好，而是要按规则注入、压缩和保护。
2. **状态与异步系统能力**：是否能处理长对话、后台记忆更新、消息压缩、并发入队等工程问题。
3. **工具治理能力**：是否能管理大量工具 schema、权限、检索召回和工具版本漂移。
4. **安全边界意识**：是否知道 Agent 调工具和执行命令前必须有 guardrails 和 sandbox。
5. **系统设计表达能力**：是否能把模块边界、数据流、失败策略讲清楚。

## 怎么用这套资料

这套资料不是按“背诵顺序”写的，而是按“面试追问深度”写的：

1. 先用本页建立项目定位，保证 30 秒、2 分钟、10 分钟都能讲。
2. 再读 `12-end-to-end-deep-dive.md`，把所有模块串成一次完整 Agent run。
3. 然后按模块读 `01` 到 `05`，补足上下文、记忆、摘要、工具、安全五个基础能力。
4. 再读 `08` 到 `11`，把当前项目的 Harness、评估、自进化、沙箱讲得更像真实工程。
5. 最后用 `06-question-bank.md` 和 `07-cheatsheet.md` 做临场复盘。

面试时不要把每篇文档都平均讲。推荐主线是：

```text
项目定位
  -> 一次 run 的完整链路
  -> 选 2-3 个最强模块深挖
  -> 用评估和观测证明有效
  -> 用 trade-off 收束
```

## 推荐讲法

### 30 秒版本

我做的是一个智能 Agent 平台，主要解决长对话和多工具场景下的稳定性问题。系统在模型调用前动态注入长期记忆、当前日期和系统提醒；对历史消息做 token 阈值压缩；对话结束后异步更新长期记忆；工具侧支持分组、延迟加载、权限控制和检索召回；所有工具调用前经过 Guardrails 做 allow/deny 决策，保证执行安全。

### 2 分钟版本

这个项目可以拆成五层：

1. **动态上下文层**：在模型调用前注入长期记忆、当前日期和系统提醒，使用隐藏消息避免污染用户真实消息。
2. **长期记忆层**：在对话后异步入队，合并同一用户/线程/Agent 的重复更新，识别纠偏和强化信号，再更新事实、摘要和冲突删除。
3. **摘要压缩层**：在上下文超过 token 阈值时，把旧消息压缩成摘要，同时保护最近消息、动态提醒和关键技能上下文。
4. **工具治理层**：工具先按配置、技能权限和分组过滤，大规模 MCP 工具不直接绑定 schema，而是通过 tool search 检索后提升。
5. **安全拦截层**：Guardrails 在工具执行前构造请求并交给 provider 判断，拒绝时返回标准化 error ToolMessage，provider 异常时默认 fail-closed。

### 10 分钟版本结构

1. 背景：Agent 长对话、多工具、多任务带来的问题。
2. 核心架构：Context / Memory / Summarization / Tools / Guardrails 五层。
3. 数据流：用户消息进入后，模型调用前注入上下文；模型调用后触发记忆；工具调用前经过权限与安全拦截。
4. 难点：避免重复注入、避免摘要破坏系统提醒、避免旧工具 schema 漂移、避免 guardrail 失败后放行危险工具。
5. 权衡：异步记忆牺牲即时一致性换吞吐；摘要牺牲原始细节换上下文窗口；延迟工具牺牲一次检索调用换 schema 预算。

## 面试表达原则

- 先讲问题，再讲设计：不要上来报类名。
- 讲清楚边界：哪个模块负责注入，哪个模块负责压缩，哪个模块负责安全拦截。
- 每个模块都准备一个 trade-off。
- 不要说“用了某某框架所以实现了”，要说“我为什么这样分层，这样做解决什么问题”。
- 遇到追问时按顺序回答：输入是什么、处理流程是什么、输出是什么、失败怎么处理。
