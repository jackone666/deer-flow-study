## 0. 项目一句话

这个项目本质上不是单纯“调用大模型 + 调工具”，而是一个面向复杂任务的 Agent Harness：它把模型、工具、上下文、记忆、子 Agent、安全治理、沙箱执行和可观测性都收敛到一套可配置、可插拔、可追踪的运行时里。

核心入口在：

- `backend/packages/harness/deerflow/agents/lead_agent/agent.py`
- `backend/packages/harness/deerflow/tools/tools.py`
- `backend/packages/harness/deerflow/agents/middlewares/`
- `backend/packages/harness/deerflow/subagents/`
- `backend/packages/harness/deerflow/guardrails/`
- `backend/packages/harness/deerflow/sandbox/`
- `backend/packages/harness/deerflow/skills/`

可以把整体链路理解成：

```text
用户请求
  -> run_agent / RunnableConfig
  -> _make_lead_agent()
  -> 解析模型、模式、agent_name、skill、tool_groups、subagent_enabled
  -> get_available_tools()
  -> skill allowed-tools 过滤
  -> deferred tool / tool_search 装配
  -> _build_middlewares()
  -> apply_prompt_template()
  -> create_agent(model, tools, middleware, system_prompt, state_schema)
  -> LangGraph 执行：LLM 节点 <-> ToolNode <-> Middleware
```

## 1. 简历项目标题

## 2. Orchestrator-Worker 多 Agent 协作模式

简历写法：

> 设计 Orchestrator-Worker 多 Agent 协作模式，由 Lead Agent 负责任务拆解、子任务委派和结果综合，Sub-Agent 独立执行检索分析、代码任务、命令操作等子任务；通过 `task` 委派工具和并发限制中间件控制单轮子代理数量，降低复杂任务执行延迟和资源失控风险。

### 2.1 Lead Agent 如何知道自己是编排者

编排逻辑首先不是写死在 Python 里，而是写进 Lead Agent 的系统提示片段。

源码：

- `_build_subagent_section()`：`backend/packages/harness/deerflow/agents/lead_agent/prompt.py:281`
- 注入时机：`apply_prompt_template()` 的 subagent 分支，相关位置在 `backend/packages/harness/deerflow/agents/lead_agent/prompt.py:974`

`_build_subagent_section()` 会根据 `max_concurrent_subagents` 动态生成提示，明确要求 Lead Agent 做三件事：

1. 拆解：把复杂任务拆成多个可并行子任务。
2. 委派：通过多个 `task` 工具调用并行启动子 Agent。
3. 综合：收集子结果后输出最终答案。

它还会把并发上限写入 prompt：

- 每轮最多 `{n}` 个 `task` 调用。
- 如果子任务数超过 `{n}`，要分批执行。
- 不能把单个简单任务包装成子 Agent。
- 对读文件、跑单条命令、需要澄清、强顺序依赖的任务，Lead Agent 应直接执行。

这解决的是 Agent 系统里很常见的问题：模型会滥用子 Agent，或者一次性开太多子 Agent，导致成本和延迟不可控。

### 2.2 `task` 工具如何把任务交给 Worker

源码：

- `backend/packages/harness/deerflow/tools/builtins/task_tool.py:1`
- `task_tool` 模块文档里已经把完整链路写清楚：主 Agent LLM 调用 `task(description, prompt, subagent_type)`，后端创建 `SubagentExecutor`，后台执行子 Agent，然后把结果作为 `ToolMessage` 返回给主 Agent。

执行链路：

```text
Lead Agent LLM
  -> tool_call: task(description, prompt, subagent_type)
  -> task_tool()
  -> SubagentExecutor.execute_async(prompt)
  -> 后台线程 / 隔离事件循环
  -> create_agent()
  -> agent.astream()
  -> 轮询任务状态
  -> SSE 推送进度
  -> 返回 Task Succeeded / Failed
  -> ToolMessage
  -> Lead Agent 综合
```

关键点：

- 子 Agent 有自己的上下文，不污染主 Agent 的上下文窗口。
- 子 Agent 可以独立使用自己的工具集和模型配置。
- 父 Agent 只拿最终结果，减少主上下文膨胀。
- 子 Agent 的 token 用量会按 `tool_call_id` 缓存，后续由 TokenUsageMiddleware 回填。

相关源码：

- token cache：`backend/packages/harness/deerflow/tools/builtins/task_tool.py:53`
- `pop_cached_subagent_usage()`：`backend/packages/harness/deerflow/tools/builtins/task_tool.py:91`
- 子 Agent 执行状态：`backend/packages/harness/deerflow/subagents/executor.py:40`
- 子 Agent 结果对象：`backend/packages/harness/deerflow/subagents/executor.py:61`
- 后台任务字典：`backend/packages/harness/deerflow/subagents/executor.py:146`
- 调度线程池：`backend/packages/harness/deerflow/subagents/executor.py:150`
- 隔离事件循环：`backend/packages/harness/deerflow/subagents/executor.py:153`

### 2.3 并发限制如何落地

Prompt 层会告诉模型不要超过并发上限，但只靠 prompt 不可靠，所以运行时还需要中间件兜底。

源码入口：

- 中间件链声明：`backend/packages/harness/deerflow/agents/lead_agent/agent.py:399`
- 启用条件：`backend/packages/harness/deerflow/agents/lead_agent/agent.py:414`
- `subagent_enabled` 读取：`backend/packages/harness/deerflow/agents/lead_agent/agent.py:737`
- `max_concurrent_subagents` 读取：`backend/packages/harness/deerflow/agents/lead_agent/agent.py:738`

简历里的“并发限制中间件控制单轮子代理数量”对应的是：

```text
Prompt 约束：告诉模型每轮最多 n 个 task
Runtime 约束：SubagentLimitMiddleware 在 after_model 阶段截断超限 task
Tool 层约束：task_tool 只负责执行被允许进入 ToolNode 的调用
```

设计价值：

- 降低多 Agent 并发失控。
- 避免模型一次性发起大量任务导致成本暴涨。
- 把“模型自觉”变成“系统约束”。

面试讲法：

> 我做的是 Orchestrator-Worker 模式。Lead Agent 只负责规划、拆解、分配和综合，Worker Agent 独立执行子任务。实现上不是简单开线程，而是通过 `task` 工具把子任务变成标准 ToolCall，由后端创建独立子 Agent 执行，并通过状态轮询、SSE 和 token 归因回传结果。并发上既有 prompt 层的行为约束，也有 `SubagentLimitMiddleware` 的运行时兜底。

简历写法：

### 3.2 为什么这算 Runtime Middleware

因为这些能力都不是业务工具本身，而是 Agent 执行过程中的横切逻辑：

- `before_agent`：在 Agent 开始前注入线程数据、上传文件、动态上下文。
- `before_model`：在模型调用前做摘要压缩、图片注入、todo 上下文。
- `wrap_model_call`：拦截模型调用，过滤 tool schema。
- `after_model`：模型返回后统计 token、限制子 Agent、检测循环。
- `wrap_tool_call`：工具执行前做安全拦截、澄清中断、guardrail 决策。
- `after_agent`：Agent 完成一轮后生成标题、写入长期记忆。

设计价值：

- 主 Agent 逻辑不需要关心每个横切能力。
- 每个能力可以独立测试。
- 增加新能力时只需要新增 middleware，并放到正确位置。
- 排查问题时能定位在具体阶段：模型前、模型后、工具前、Agent 后。

面试讲法：

> 我把 Agent Runtime 设计成中间件链，而不是把所有逻辑堆在主循环里。比如上下文注入在 `before_agent`，摘要压缩在 `before_model`，工具治理在 `wrap_model_call` 和 `wrap_tool_call`，token 归因在 `after_model`，记忆写入在 `after_agent`。这样主循环保持稳定，新增能力都是插件式接入，问题定位也能按阶段拆开。

## 4. Context Engineering Layer（上下文工程层）

简历写法：

> 设计 Context Engineering Layer，在模型调用前统一管理长期记忆、当前日期、系统提醒、运行态上下文和历史摘要；通过隐藏消息注入与静态 system prompt 分离，降低上下文污染风险并提升前缀缓存复用能力。

这个亮点主要由两个模块组成：

- `DynamicContextMiddleware`
- `DeerFlowSummarizationMiddleware`

### 4.1 动态上下文为什么不直接拼进 system prompt

源码：

- `backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:1`
- `DynamicContextMiddleware`：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:61`

文件头部已经说明了设计目的：

```text
系统提示保持完全静态，以最大化跨用户/跨会话的前缀缓存复用。
当前日期始终注入。
memory.injection_enabled 为 True 时，按用户维度注入记忆。
```

如果把“当前日期、用户长期记忆、运行态提醒”都拼进 system prompt，那么每个用户、每一天、每次记忆变化都会让 system prompt 变化，模型 prefix cache 命中率会下降。

所以这里采用的是：

```text
静态 system prompt：角色、规则、工具策略、子 Agent 规则
动态 hidden HumanMessage：当前日期、长期记忆、system-reminder
```

### 4.2 隐藏消息怎么注入

关键源码：

- 构建完整 reminder：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:89`
- 注入 `current_date`：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:100`
- 生成隐藏 HumanMessage：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:123`
- 标记 `hide_from_ui` 和 `dynamic_context_reminder`：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:137`
- 首轮注入逻辑：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:168`
- 同一天不重复注入：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:183`
- 跨午夜只注入轻量日期更新：`backend/packages/harness/deerflow/agents/middlewares/dynamic_context_middleware.py:187`

实现细节：

1. 首轮找到第一条真实用户消息。
2. 构造 `<system-reminder>`，里面包含 memory 和 `<current_date>`。
3. 创建一个隐藏 `HumanMessage`，带 `hide_from_ui=True`。
4. 使用 ID 交换技巧：reminder 复用原消息 ID，原用户消息改成派生 ID。
5. 后续同一天不重复注入，避免上下文不断膨胀。
6. 跨午夜时只补一条日期更新，不重新注入整段记忆。

### 4.3 `_preserve_dynamic_context_reminders` 是做什么的

源码：

- `backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:194`

它的作用是：摘要压缩时，不要把动态上下文提醒吞进 summary 里。

原因：

如果隐藏的 `<system-reminder>` 被当成普通历史消息压缩掉，后续 `DynamicContextMiddleware` 可能会误判“从未注入过动态上下文”，然后把新的 reminder 注入到错误位置，例如注入到 summary 消息前面，导致上下文顺序混乱。

所以 `_preserve_dynamic_context_reminders()` 会：

```text
messages_to_summarize 中找到 dynamic_context_reminder
  -> 从待摘要列表移除
  -> 放到 preserved_messages 前面
```

这是一处很适合面试展开的细节，因为它体现了上下文工程不只是“把文本塞进去”，还要考虑消息生命周期、摘要压缩、副作用和模型缓存。

### 4.4 摘要压缩如何处理历史上下文

源码：

- 同步摘要入口：`backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:131`
- 异步摘要入口：`backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:158`
- token 阈值判断：`backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:136`
- 切分消息：`backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:140`
- 构造 summary message：`backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:185`
- 技能上下文抢救：`backend/packages/harness/deerflow/agents/middlewares/summarization_middleware.py:212`

摘要流程：

```text
before_model
  -> 统计当前 messages token
  -> 判断是否达到压缩阈值
  -> 决定 cutoff_index
  -> 划分待摘要消息 / 保留消息
  -> 恢复最近加载的 skill bundle
  -> 保留 dynamic context reminder
  -> 调用 summary model
  -> RemoveMessage(REMOVE_ALL_MESSAGES)
  -> summary HumanMessage(name="summary") + preserved_messages
```

`name="summary"` 的意义：

- 前端可以隐藏它。
- 模型仍然可以看到它。
- 它不会被误认为真实用户消息。

面试讲法：

> 我把 Context Engineering 分成静态 prompt 和动态上下文两层。静态 prompt 放稳定规则，动态内容通过隐藏 HumanMessage 注入，这样用户记忆和日期不会破坏 prefix cache。摘要压缩前还会把动态 reminder 和最近加载的 skill bundle 从摘要区抢救出来，避免模型在长对话里丢掉关键上下文。

## 5. Agent Memory Pipeline（记忆管道）

简历写法：

> 建设 Agent Memory Pipeline，支持对话后异步入队、去抖合并、用户纠偏识别、强化信号检测、事实抽取、记忆摘要更新和冲突事实删除，实现从对话事件到长期记忆的闭环沉淀。

### 5.1 记忆管道整体流程

主要源码：

- `MemoryMiddleware`：`backend/packages/harness/deerflow/agents/middlewares/memory_middleware.py`
- `MemoryUpdateQueue`：`backend/packages/harness/deerflow/agents/memory/queue.py:59`
- `MemoryUpdater`：`backend/packages/harness/deerflow/agents/memory/updater.py`

核心流程：

```text
Agent 一轮结束

  -> LLM 抽取、更新、删除冲突事实

```

### 5.2 去抖合并如何实现

源码：

- 队列设计说明：`backend/packages/harness/deerflow/agents/memory/queue.py:1`
- 合并键：`backend/packages/harness/deerflow/agents/memory/queue.py:73`
- 普通入队：`backend/packages/harness/deerflow/agents/memory/queue.py:82`
- 立即入队：`backend/packages/harness/deerflow/agents/memory/queue.py:138`
- 合并逻辑：`backend/packages/harness/deerflow/agents/memory/queue.py:165`
- timer 调度：`backend/packages/harness/deerflow/agents/memory/queue.py:206`

合并键是：

```python
(thread_id, user_id, agent_name)
```

这个设计说明记忆不是全局乱写，而是按线程、用户、自定义 Agent 隔离。

去抖的价值：

- 用户连续对话时，不会每句话都触发一次 LLM 记忆更新。
- 30 秒窗口内同一个 `(thread_id, user_id, agent_name)` 会覆盖旧消息，保留最新上下文。
- correction / reinforcement 标记用逻辑或合并，避免关键纠偏信号丢失。

### 5.3 为什么有 `add_nowait`

`add()` 是去抖延迟处理，`add_nowait()` 是立即后台处理。

适用场景：

- 长对话触发摘要压缩前，需要先把即将被压缩掉的重要消息冲入长期记忆。
- 如果仍然等 debounce，可能历史消息已经被 summary 替换，记忆抽取材料不完整。

源码：

- `add_nowait()`：`backend/packages/harness/deerflow/agents/memory/queue.py:138`

这里有一个很工程化的细节：记忆更新虽然可能从异步上下文触发，但最终委托到同步 `model.invoke()`，必要时放到线程池执行。

面试讲法：

> 记忆不是直接在主链路里同步写的，而是 after_agent 后进入 MemoryUpdateQueue。队列按 `(thread_id, user_id, agent_name)` 做隔离和去抖合并，减少频繁 LLM 调用。真正更新时由 MemoryUpdater 构造记忆更新 prompt，抽取事实、处理纠偏和冲突，再写入 storage。为了避免异步连接池跨 event loop 的问题，记忆更新最终走同步模型调用并按需 offload 到线程池。

## 6. 对话摘要压缩中间件

简历写法：

> 通过对话摘要压缩中间件，在长上下文任务中自动压缩历史消息并保留关键状态，降低长会话 token 成本和模型遗忘风险。

这个点和 Context Engineering 有交叉，但可以单独讲成“长上下文治理”。

### 7.4 Deferred Tool Filter：大规模 MCP 工具上下文压缩

源码：

- `DeferredToolFilterMiddleware`：`backend/packages/harness/deerflow/agents/middlewares/deferred_tool_filter_middleware.py:24`
- 模型调用前过滤 schema：`backend/packages/harness/deerflow/agents/middlewares/deferred_tool_filter_middleware.py:54`
- 未提升工具调用拦截：`backend/packages/harness/deerflow/agents/middlewares/deferred_tool_filter_middleware.py:66`

设计背景：

MCP 生态下工具可能非常多。如果每个工具 schema 都直接绑定给模型，会带来几个问题：

- system/tool schema 上下文膨胀。
- 模型选择工具时噪音变大。
- 工具 schema 变动影响 prompt cache。
- 大量不相关工具增加误调用概率。

实现策略：

```text
ToolNode：仍然持有完整工具，保证运行时可执行
LLM bind_tools：只暴露活跃工具 + 已 promoted 工具
tool_search：让模型按需搜索工具
DeferredToolFilterMiddleware：隐藏未 promoted 的 schema
wrap_tool_call：如果模型绕过规则调用未 promoted 工具，返回错误 ToolMessage
```

`catalog_hash` 的作用：

- 用来标识当前工具目录版本。
- 如果工具目录变化，之前 promoted 的工具不能无条件复用。
- 避免旧工具提升状态污染新工具集合。

### 7.5 Guardrails：工具执行前授权

源码：

- `GuardrailMiddleware`：`backend/packages/harness/deerflow/guardrails/middleware.py:20`
- 构造授权请求：`backend/packages/harness/deerflow/guardrails/middleware.py:44`
- 同步工具拦截：`backend/packages/harness/deerflow/guardrails/middleware.py:81`
- 异步工具拦截：`backend/packages/harness/deerflow/guardrails/middleware.py:114`
- 拒绝消息：`backend/packages/harness/deerflow/guardrails/middleware.py:60`

执行逻辑：

```text
工具调用请求
  -> GuardrailMiddleware.wrap_tool_call
  -> 构造 GuardrailRequest(tool_name, tool_input, agent_id, timestamp)
  -> provider.evaluate()
  -> allow=True: 调用 handler 执行真实工具
  -> allow=False: 返回 ToolMessage(status="error")
  -> provider 异常:
       fail_closed=True  -> 拒绝
       fail_closed=False -> 放行并记录告警
```

这对应简历中的：

- allow/deny 决策。
- fail-closed。
- 拒绝原因返回。
- 工具错误标准化。

面试讲法：

> Tool Governance 不是一个点，而是一层工具运行时治理。工具来源先统一装配，再经过 group、skill allowed-tools、deferred tool、guardrail 多层过滤。MCP 工具不会一次性全部暴露给模型，而是通过 tool_search 按需提升；工具真正执行前还会进入 GuardrailMiddleware 做 allow/deny 决策。这样可以同时解决上下文膨胀、误调用和越权调用问题。

### 8.2 Bash 命令审计如何做

- 高危规则：`backend/packages/harness/deerflow/agents/middlewares/sandbox_audit_middleware.py:53`

- 审计日志：`backend/packages/harness/deerflow/agents/middlewares/sandbox_audit_middleware.py:258`

分类规则：

- `block`：高危，直接拒绝执行，例如 `rm -rf /`、`curl | bash`、`dd if=`、`LD_PRELOAD`、fork bomb。
- `warn`：中危，执行但追加警告，例如 `pip install`、`chmod 777`、`sudo`、修改 `PATH`。
- `pass`：安全，正常执行。

工程细节：

- 先对完整命令做高危扫描，捕获跨语句攻击。
- 再按 `&&`、`||`、`;` 拆分复合命令。
- 拆分时识别单双引号，避免把引号内的符号当操作符。
- 未闭合引号会失败关闭，按整条命令处理。
- 每次 bash 调用都写结构化 JSON 审计日志。

## 11. 面试追问模板

### 11.1 如果问“你们这个 Agent 项目和普通 ChatBot 有什么区别”

可以答：

> 普通 ChatBot 主要是一次模型调用加少量工具，而这个项目是 Agent Harness。它有运行时工厂负责模型和工具装配，有 middleware 管理上下文、摘要、安全和观测，有 Tool Governance 控制大规模工具暴露和授权，有 Memory Pipeline 做长期记忆沉淀，还有 Orchestrator-Worker 模式支持复杂任务并行拆解。所以它关注的是生产化 Agent 的稳定性、可控性和可观测性。

### 11.2 如果问“多 Agent 是怎么实现的”

可以答：

> Lead Agent 的 prompt 会在 subagent_enabled 时注入编排规则，要求它把复杂任务拆成多个子任务，并用 `task` 工具委派。`task_tool` 收到 tool call 后创建 `SubagentExecutor`，在后台线程和隔离事件循环里启动独立子 Agent，子 Agent 完成后结果作为 ToolMessage 回给 Lead Agent。并发上限由 prompt 和 SubagentLimitMiddleware 双层控制，token 用量按 tool_call_id 回填到父 Agent。

### 11.3 如果问“上下文工程做了什么”

可以答：

> 我们把 system prompt 做成静态的，动态内容比如当前日期和长期记忆通过隐藏 HumanMessage 注入。这样不同用户的记忆不会污染 system prompt，也能提升 prefix cache 复用。长对话里用 SummarizationMiddleware 在模型调用前压缩历史，同时保留 dynamic context reminder 和最近加载的 skill bundle，避免摘要导致关键上下文丢失。

### 11.4 如果问“Tool Governance Layer 是什么意思”

可以答：

> 它是工具从加载到执行的治理层。加载时统一合并 config tools、built-in tools、MCP tools、ACP tools、subagent task tools；暴露时按 tool_groups 和 skill allowed-tools 过滤；对 MCP 这类大规模工具用 deferred tool search 按需提升 schema；执行前通过 GuardrailMiddleware 做 allow/deny 决策，并且 fail-closed。目标是降低上下文膨胀、工具误调用和越权调用。

### 11.5 如果问“记忆系统怎么避免乱写”

可以答：

> 记忆写入不在主链路里同步执行，而是在 after_agent 后进入 MemoryUpdateQueue。队列按 `(thread_id, user_id, agent_name)` 隔离，使用 debounce 合并高频对话，纠偏和强化信号用逻辑或保留。真正更新时由 MemoryUpdater 构造 prompt 让 LLM 抽取事实、更新摘要和删除冲突事实，最后写入 storage。下轮再由 DynamicContextMiddleware 注入。

### 11.6 如果问“沙箱和安全怎么做”

可以答：

> 工具执行面通过 sandbox tools 进入受控环境。对 bash 这类高风险工具，我加了 SandboxAuditMiddleware，在执行前先做命令分级。高危命令直接 block，中危命令执行但追加 warning，安全命令正常执行。分类时先扫完整命令，再拆复合命令，并处理引号和异常情况；每次调用都会写结构化审计日志。

### 11.7 如果问“可观测性怎么做”

可以答：

> Agent 工厂会把 agent_name、model_name、subagent_enabled、tool_groups、available_skills 等写入 trace metadata。TokenUsageMiddleware 会给每个 AI step 标注 attribution，例如 subagent_dispatch、tool_batch、final_answer，并按 tool_call_id 关联工具调用。子 Agent 的 token 用量会从 task_tool 缓存里取出，再合并回父 Agent 发起 task 的 AIMessage。

## 12. 建议背诵版

30 秒版：

> 我这个 Agent 项目核心是一个生产化 Agent Harness。Lead Agent 工厂负责统一装配模型、工具、prompt、中间件和状态；Runtime Middleware 负责上下文、摘要、安全、记忆和观测；Tool Governance 负责 MCP 等大规模工具的按需暴露和授权；复杂任务通过 `task` 工具委派给子 Agent 并行执行；长期记忆通过去抖队列和 LLM 更新器沉淀。整体目标是让 Agent 在复杂任务下可扩展、可控、可追踪。

2 分钟版：

> 实现上，入口是 `_make_lead_agent()`。它从运行时配置里解析模型、计划模式、子 Agent 开关、自定义 Agent、用户和技能信息，然后调用 `get_available_tools()` 装配 built-in、MCP、ACP、sandbox 和 subagent 工具，再经过 skill allowed-tools 和 deferred tool search 过滤，最后把工具、中间件、system prompt 和 state schema 交给 LangGraph `create_agent`。
>
> 多 Agent 部分是 Orchestrator-Worker。Lead Agent 的系统提示会在 `subagent_enabled` 时注入编排规则，要求它拆解、委派、综合；真正执行时通过 `task` 工具创建独立子 Agent，在后台线程和隔离 event loop 里运行，并通过 ToolMessage 返回结果。并发上限既写在 prompt 里，也由 SubagentLimitMiddleware 兜底。
>
> 上下文层做了静态 system prompt 和动态 hidden HumanMessage 分离，把当前日期和长期记忆以 `<system-reminder>` 注入，提升 prefix cache 复用；长会话用 SummarizationMiddleware 自动压缩历史，并保留动态 reminder 和最近技能上下文。安全和治理层则通过 deferred tool、Guardrails、SandboxAuditMiddleware 控制工具暴露和执行风险。观测层通过 trace metadata、step attribution 和 tool_call_id 把模型调用、工具调用、子 Agent token 串起来。
