## 第一部分：基础 10 题（任意 Agent 项目都该懂）

### Q01: DeerFlow 主子智能体怎么通信的？

**30 秒：** 通过 `task` 工具桥接。LLM 调用 `task(description, prompt, subagent_type)` → `task_tool` 在后台启动独立 LangGraph Agent → 5 秒轮询等待结果 → 子 Agent 完成后结果作为 ToolMessage 回流给主 Agent。**没有独立消息通道，所有通信通过工具调用机制完成。**

**完整：**
1. 主 Agent 的 system prompt 里告诉 LLM "你有 task 工具，可以委派复杂子任务"
2. LLM 输出 `AIMessage(tool_calls=[{name: "task", args: {description, prompt, subagent_type}, id: "call_xxx"}])`
3. LangGraph 框架路由到 `task_tool` 实现函数
4. `task_tool` 解析 SubagentConfig → 提取共享 state（sandbox / thread_data 引用传递）→ 创建 SubagentExecutor → `execute_async` 启动后台任务
5. `execute_async` 提交到 ThreadPoolExecutor(max_workers=3)，scheduler 线程把协程提交到持久 daemon event loop
6. 子 Agent 在 daemon loop 里跑独立 LangGraph 图，独立消息历史 / 独立 token 桶
7. 主 Agent 的 `task_tool` 进入 5 秒轮询循环，期间通过 SSE 推送 task_running 事件
8. 子 Agent 完成 → `try_set_terminal(COMPLETED)` → 主 Agent 轮询命中 → cache token / report journal / SSE task_completed → return "Task Succeeded. Result: ..."
9. 框架把字符串包装成 ToolMessage → 加到主 Agent state.messages → 主 Agent 下一轮 LLM 看到结果继续推理

### Q02: task 工具是什么？为什么要放在工具列表里？

**30 秒：** `task` 是 DeerFlow 唯一注册成 LangChain `@tool` 的"委派工具"。LLM 调用它就等于 fork 子 Agent。放在工具列表里是为了让**LLM 自主决定何时 fork**，不是开发者预先定义路由。

**完整：**
- 文件：`tools/builtins/task_tool.py:312`，装饰器 `@tool("task", parse_docstring=True)`
- 通过 `tools/tools.py:SUBAGENT_TOOLS = [task_tool]` 在 `subagent_enabled=True` 时加入主 Agent 工具列表
- 子 Agent 拿到的工具列表里**没有 task**（`get_available_tools(subagent_enabled=False)` 过滤掉）—— 这是递归防护
- 设计哲学：把 fork 决策权交给 LLM，让它根据任务复杂度自己判断

### Q03: SubagentExecutor 是怎么跑子 Agent 的？

**30 秒：** 三步走：① 用 `langchain.agents.create_agent` 创建独立 LangGraph 图 ② 构造初始 state（含 system_prompt + 任务 + 共享 sandbox）③ 在持久 daemon event loop 里 `agent.astream(state)`，每个 chunk 边界检查 cancel_event 和捕获 AIMessage。

**完整：**
- 入口：`SubagentExecutor.execute_async(task, task_id)` → 提交 ThreadPool → `_aexecute(task, result_holder)`
- 关键代码 `executor.py:474-668`：
  - `_build_initial_state` 构造 state（包含技能加载）
  - `_create_agent` 用 `create_agent(model, tools, middleware, state_schema=ThreadState)` 创建子图
  - `SubagentTokenCollector` 注入到 callbacks 收集 token
  - `agent.astream(state, config=run_config)` 流式执行
  - 每个 chunk：检查 `cancel_event` → 捕获 AIMessage → 累加 ai_messages
  - 完成后 `_extract_final_result` 取最后一条 AIMessage 的 text，调用 `try_set_terminal(COMPLETED, ...)`

### Q04: 主 Agent 和子 Agent 的中间件链一样吗？

**30 秒：** 不一样。主 Agent 19 个，子 Agent 9-10 个（精简版）。子 Agent **没有** Summarization / TokenUsage / SubagentLimit / Clarification / Title / Memory —— 这些都是主对话特性，子 Agent 不需要。

**完整：**
- 主 Agent 中间件由 `_build_middlewares` 组装（`agents/lead_agent/agent.py:373`）
- 子 Agent 中间件由 `build_subagent_runtime_middlewares` 组装（`tool_error_handling_middleware.py`，executor.py:355 调用）
- 子 Agent **不挂载**：DynamicContext / Summarization / Todo / TokenUsage / Title / Memory / DeferredToolFilter / SubagentLimit / LoopDetection / SafetyFinish / Clarification
- 子 Agent **挂载**：ThreadData / Uploads / Sandbox / DanglingToolCall / LLMError / Guardrail / SandboxAudit / ToolError / ViewImage（如果模型支持视觉）

### Q05: SubagentResult 里的 cancel_event 是干嘛的？

**30 秒：** `threading.Event`，用于跨线程的协作式取消信号。父 Agent 调 `request_cancel_background_task(task_id)` → `result.cancel_event.set()` → 子 Agent 在 astream 每个 chunk 边界检查到 → 主动退出。

**完整：** 见第 8 章。关键点：
- `threading.Event` 不是 `asyncio.Event` —— 因为要跨线程跨事件循环
- 检查时机是 chunk 边界（不能强中断 LLM 调用 / 工具调用）
- 所以"取消延迟"最坏 = 当前 chunk 跑完时间（几秒到几十秒）

### Q06: 为什么用 ThreadPoolExecutor 而不是 asyncio.gather？

**30 秒：** ThreadPoolExecutor 给每个子 Agent 一个独立线程，**生命周期与父协程解耦** —— 父协程被取消时子 Agent 还能跑（或优雅退出）。`asyncio.gather` 会让父协程等所有子协程，且 cancel 会立即穿透。

**完整：**
- ThreadPool 的 worker 跑同步函数 `run_task()`，内部 `Future.result(timeout=...)` 阻塞等待协程完成
- 协程在持久 daemon loop 里跑，daemon loop 不属于父协程的事件循环
- 这样设计的好处：父协程异常退出不会撕裂 daemon loop 上的协程，子 Agent 可以走完 try/finally 清理资源
- 如果用 `asyncio.gather`，父子协程绑死在同一个 loop 上，cancel 一个就 cancel 全部

### Q07: 子 Agent 能不能再 fork 孙 Agent？

**30 秒：** 默认不能。`task_tool.py:434` 在调用 `get_available_tools` 时传 `subagent_enabled=False`，子 Agent 拿到的工具列表里没有 `task` 工具，所以 LLM 想 fork 也调不到。

**完整：**
- 这是显式的递归防护
- 改起来要两处：① 子 Agent config 的 `disallowed_tools` 移除 `"task"` ② `task_tool.py:434` 改 True
- 但**强烈不建议**：递归无约束 → 容易爆 token / 爆线程池 / token 计费链路只算一层

### Q08: SubagentConfig 的 9 个字段都是什么？

**30 秒：** `name` / `description` / `system_prompt` / `tools`（白名单）/ `disallowed_tools`（黑名单）/ `skills` / `model`（"inherit"）/ `max_turns`（recursion_limit，50）/ `timeout_seconds`（900）。

**完整：** 见第 5 章 §2.2 表格。重点：
- `tools=None` 含义"继承父全部工具"（不是禁用）
- `skills` 三态：`None`（继承所有）/ `[]`（禁用）/ `["a", "b"]`（白名单）
- `model="inherit"` → 沿用父模型 → `resolve_subagent_model_name` 三层回退

### Q09: `subagent_type` 怎么解析？

**30 秒：** `registry.get_subagent_config(name)` 三层查找：① 内置（BUILTIN_SUBAGENTS）② config.yaml 的 custom_agents ③ config.yaml 的 agents 段覆盖。找不到返回 None，task_tool 返回错误字符串给 LLM。

**完整：** 见第 5 章 §3.1。优先级：
- timeout / max_turns：per-agent override > 全局默认（仅内置）> config 自己的值
- model / skills：仅 per-agent override（无全局默认）

### Q10: 子 Agent 完成后结果怎么回到主 Agent？

**30 秒：** `task_tool` 在轮询循环里检测到 COMPLETED，return `f"Task Succeeded. Result: {result.result}"` 字符串。LangGraph 框架自动包装成 `ToolMessage(content=..., tool_call_id="call_xxx")`，加到 state.messages。主 Agent 下一轮 LLM 看到这条消息继续推理。

**完整：**
- 字符串规范："Task Succeeded. Result: {text}" / "Task failed. Error: {err}" / "Task cancelled by user." / "Task timed out. Error: {err}"
- LangGraph 在 ToolNode 内部把字符串塞 ToolMessage 的 content
- `tool_call_id` 一一对应 AIMessage 里的 tool_call id —— 主 Agent 知道是哪个 task 的结果
- token_usage 走带外通道（`_subagent_usage_cache`）不是塞在字符串里

## 第二部分：进阶 10 题（DeerFlow 特色）

### Q11: 持久 daemon event loop 解决什么问题？

**30 秒：** 父 Agent 在 async 上下文里跑（agent.astream），子 Agent 也要 async 跑。**新建 loop 会撕裂 httpx / MCP 客户端的连接池**（这些连接绑定到创建它们的 loop）。所以全局开一个 daemon 线程跑 `loop.run_forever()`，所有子 Agent 共享。

**完整：** 见第 4 章 §4。关键：
- `_get_isolated_subagent_loop()` 惰性创建（健康检查不通过就重建）
- 用 `asyncio.run_coroutine_threadsafe` 跨线程提交协程
- `daemon=True` + `atexit.register` 双重清理

### Q12: 协作式取消 vs 抢占式取消，DeerFlow 选哪个？为什么？

**30 秒：** 协作式。`Future.cancel()` 杀不死正在跑的同步代码（如 bash 工具调用）。`asyncio.CancelledError` 会撕裂 httpx 连接池。所以用 `threading.Event` + 子 Agent 在 chunk 边界主动检查的方案 —— 代价是延迟，收益是干净退出。

**完整：** 见第 8 章。

### Q13: Token 三层合并是哪三层？

**30 秒：** ① `SubagentTokenCollector`（LangChain 回调，子 Agent 内累加 records）② `_subagent_usage_cache`（全局 dict，按 tool_call_id 索引的带外通道）③ `TokenUsageMiddleware.after_model`（pop cache 合并到调度它的 AIMessage.usage_metadata）。

**完整：** 见第 7 章。RunJournal 是另一条独立链路（按 caller 分桶审计）。

### Q14: SubagentLimitMiddleware 为什么钳制到 [2, 4]？

**30 秒：** 下界 2：1 个就没必要 fork。上界 4：① ThreadPool 只有 3 个 worker，超 3 排队 ② LLM API 通常有 RPS 限制 ③ 前端展示 4+ 任务流混乱。`_clamp_subagent_limit` 强制钳制范围，引导用户走合理配置。

**完整：** 见第 9 章 §4.3。

### Q15: 三层并发防护各自防什么？

**30 秒：** ① Prompt 软约束防 LLM 任性输出 ② Middleware 硬截断防 prompt 失效 ③ ThreadPool 物理上限防 Middleware 失效。三层独立，每层假设其他层失败。

**完整：** 见第 9 章。

### Q16: `_subagent_usage_cache` 为什么必须用 pop 不用 get？

**30 秒：** 两个理由：① 防内存泄漏（不 pop dict 越来越大）② 防重复合并（state 回放时 get 会重复加 token）。每个 tool_call_id 只 pop 一次保证语义正确。

**完整：** 见第 7 章 §3.5。

### Q17: TokenUsageMiddleware 的反向遍历算法做了什么？

**30 秒：** 从 messages 末位往前找连续的 ToolMessage（每个对应一个完成的 task）→ 每个 ToolMessage pop 出 cache 里的 usage → 再往前找包含这个 tool_call_id 的 AIMessage → 合并 usage_metadata。多个并发 task 都映射到同一条 AIMessage 上累加。

**完整：** 见第 7 章 §4。重点：用 `state_updates` dict 累加，避免覆盖；用 `model_copy(update=...)` 不修改原对象。

### Q18: 父子 Agent 怎么共享沙箱和文件？

**30 秒：** 引用传递。`task_tool.py:391-392` 中 `sandbox_state = runtime.state.get("sandbox")` 拿到的是 dict 引用，**直接传给 SubagentExecutor**，子 Agent 改了主 Agent 看得见。`thread_data` 同理。

**完整：**
- `runtime.state` 是 LangGraph state dict
- `sandbox` 字段是 `SandboxState` TypedDict（包含 sandbox_id / mounts / runtime info）
- 子 Agent 在 `_build_initial_state` 里把这个 ref 直接放进自己的 state["sandbox"]
- 文件层面：父子写入 `/mnt/user-data/workspace/foo.txt`，物理上是同一个文件（沙箱挂载点共享）

### Q19: 技能白名单是怎么父子合并的？

**30 秒：** 取交集。`_merge_skill_allowlists(parent, child)`：父 None → 沿用子；子 None → 限制为父；都有 → 取交集。父的限制是绝对边界，子无法突破。

**完整：** 见第 3 章 §5.2 真值表。

### Q20: 为什么 trace_id 要在 task_tool 里生成而不是 SubagentExecutor 里？

**30 秒：** 因为 trace_id 要**关联父子日志**。task_tool 在父协程上下文里能拿到父 Agent 的 trace_id（`metadata.get("trace_id")`），传给 Executor 让子 Agent 用同一个 ID。如果父没有 trace_id 就生成新的（`uuid.uuid4()[:8]`）。

**完整：**
- 父 Agent 的 trace_id 来自请求级 metadata
- 子 Agent 的 logger 都用 `[trace={self.trace_id}]` 前缀
- 这样查日志时一个 trace_id 能拿到完整的父子调用链

## 第三部分：系统设计 5 题

### Q21: 让你重新设计一个主子 Agent 系统，会怎么做？

**完整回答：**

**架构选择：**
1. **通信机制**：保留"工具调用即委派"模式 —— 优点是主子完全解耦，子 Agent 失败不影响父；缺点是同步阻塞，但可通过 SSE 旁路推送中间状态弥补。
2. **执行隔离**：保留持久 daemon loop —— 解决 httpx/MCP 连接池绑定问题。如果跨进程隔离需求强，可加 multiprocessing 但要序列化所有共享状态（成本高）。
3. **并发限制**：三层防护（Prompt + Middleware + ThreadPool）—— 我会**额外加一层**：基于实际下游 LLM 限流的自适应调整（监控 RPS / TPM 自动降级到 max_concurrent=2）。
4. **取消机制**：协作式 + threading.Event + chunk 边界检查 —— 保留。**额外**：在工具实现里加细粒度 timeout（比如 bash 30 秒），减少协作延迟。
5. **Token 追踪**：双链路（消息维度 + 审计维度）—— 保留。**额外**：把 `_subagent_usage_cache` 换成带 TTL 的 LRU（防止异常路径残留）。

**取舍：**
- 同步阻塞是接受的代价 —— 因为前端可以用 SSE 看到中间过程
- 单进程多线程是接受的限制 —— 简化共享状态管理，扩展性靠多 worker 横向

### Q22: 如何让这套系统支持跨进程 / 跨机器扩展？

**完整回答：**

需要改的地方：

1. **状态共享**：`_background_tasks` / `_subagent_usage_cache` 改用 Redis（key=task_id / tool_call_id，value=序列化后的 SubagentResult / usage dict）
2. **沙箱共享**：`sandbox_state` / `thread_data` 不能引用传递，改为：① 共享文件系统（NFS / S3）② 状态序列化到 Redis
3. **取消信号**：`cancel_event` 改用 Redis Pub/Sub 或专门的 cancel topic
4. **调度器**：ThreadPoolExecutor 换成分布式任务队列（Celery / RQ / Temporal）
5. **追踪**：所有 trace_id 串到 OpenTelemetry / Langfuse，保证跨进程关联

**新挑战：**
- 序列化开销（每次状态读取要 pickle/json）
- 一致性问题（多 worker 同时改 state 要分布式锁）
- 故障域扩大（Redis 挂了整个系统瘫）

**结论：** DeerFlow 当前单进程方案适合中小规模（几十 RPS），跨机扩展前先评估是否真有必要。

### Q23: 如何给系统加水平扩展能力？

**完整回答：**

**方案 A：多 worker 单实例**
- 启动 N 个 worker 进程（FastAPI + uvicorn `--workers N`）
- 每个 worker 独立的 ThreadPool / daemon loop
- thread_id sticky 路由（同一会话进同一 worker）—— Nginx ip_hash 或 LB sticky session
- 优点：实现简单
- 缺点：worker 间状态不共享，user 不能跨 worker

**方案 B：分布式调度**
- Lead Agent 在 worker 里跑（无状态）
- 子 Agent 调度到独立的 worker pool（k8s job / Celery worker）
- task_tool 改为提交到分布式队列，轮询从队列取结果
- 优点：水平扩展能力强
- 缺点：实现复杂，序列化成本

DeerFlow 当前是方案 A 的简化版（单 worker 多线程）。如果要做方案 B，建议从 task_tool 这一层切入 —— 把 `executor.execute_async` 替换成"提交到 Celery 队列"，最少改动。

### Q24: 如何加一个"广告 Agent"专用子 Agent？

**完整回答：**

**步骤：**

1. **定义 SubagentConfig**（在 config.yaml 或 Python 代码）：

**注意点：**

- `skills` 可以加载特定的诊断知识库

### Q25: 系统支持的最大并发用户数 / 最大并发任务数是多少？

**完整回答：**

**当前架构上限：**
- 单 worker：3 个并发子 Agent（ThreadPool max_workers=3）
- 主 Agent 协程数：受 FastAPI / uvicorn 的 worker 数决定（默认 1 个 worker，单 loop 几百到几千协程都可以）
- 实际瓶颈：LLM API 限流（OpenAI 默认 5000 RPM 个人 / 60000 RPM 企业）

**算法：**
- 假设每个用户对话平均产生 1 个 task → 3 个并发子 Agent ≈ 3 个并发用户
- 假设每次 LLM 调用 5 秒 → 单 worker 5 RPS → 50000 daily 用户（假设每人每天 100 次对话）

**扩展路径：**
- 加 worker：N 个 worker × 3 并发 = 3N 并发任务
- 加更高 LLM 配额
- 大用户量时拆服务（Lead Agent / Subagent / Tools 各自部署）

DeerFlow 当前定位：**几十到几百并发用户**的中等规模 SaaS，更大规模需要架构改造。

## 第四部分：故障排查 5 题

### Q26: 子 Agent 完成了但前端 token 数没更新，怎么排查？

**排查链路：**

```
1. 看 SSE 事件：前端是否收到 task_completed 事件？
   ├─ 收到 → 走 Layer 2/3
   └─ 没收到 → task_tool 没执行到 writer({"type": "task_completed"...})
                 检查 task_tool 是否正常 return（看日志 [trace=xxx] Task xxx completed）

2. 看 _subagent_usage_cache：
   ├─ 在 task_tool.py:521 加日志 print(f"cached: {tool_call_id}={usage}")
   └─ 验证 cache 是否写入

3. 看 TokenUsageMiddleware：
   ├─ token_usage_middleware.py:300 加日志 print(f"pop: {tool_msg.tool_call_id}")
   ├─ 是否 pop 出 usage？
   └─ 是否找到对应 AIMessage？

4. 看 AIMessage.usage_metadata：
   └─ 在 state["messages"] 里 dump，看最后一条 AIMessage 的 usage_metadata
```

**常见原因：**
- `token_usage.enabled=False` → `_cache_subagent_usage` 跳过写入
- 子 Agent 异常退出，token_usage_records 为空
- TokenUsageMiddleware 没挂载（`token_usage.enabled=False`）

### Q27: 子 Agent 卡在 RUNNING 永远不结束，怎么办？

**排查：**

1. **看 trace 日志**：找 `[trace={trace_id}] Subagent xxx ...` 系列
2. **看 ai_messages**：`result.ai_messages` 最后一条是什么？
   - LLM 调用？→ LLM API 卡了
   - 工具调用？→ 看具体工具
3. **检查 cancel_event**：`request_cancel_background_task(task_id)` 后是否退出？
   - 不退出 → 子 Agent 卡在工具调用里（cancel_event 边界检查未触发）

**常见原因：**
- bash 工具调用没 timeout，跑了 60+ 秒
- httpx 调用没 timeout，等响应
- 工具内部死循环

**临时缓解：**
- 调短 `timeout_seconds`（默认 900 → 600）让线程池超时早触发
- 给关键工具加 timeout（subprocess.run(timeout=30)）

### Q28: 主 Agent 一次响应吐了 8 个 task 怎么办？

**正常流程：**
- LLM 输出 8 个 task → SubagentLimitMiddleware.after_model 截断到 3 个 → 3 个并发执行 → 3 个 ToolMessage 回流
- LLM 下一轮看到 3 个结果 + 系统约束（"被丢弃 5 个"在 prompt 里有提示）→ 续发剩下 5 个

**如果 8 个全部启动：**
- 检查 `subagent_enabled` 是否 True
- 检查 SubagentLimitMiddleware 是否在 middlewares 列表里
- 检查 `max_concurrent_subagents` 配置
- 看日志有没有 "Truncated X excess task tool call(s)..." warning

**Layer 3 兜底：**
- 即使 8 个 tool_call 全部进入 ToolNode，每个 task 调用 `executor.execute_async` → ThreadPool（max_workers=3）→ 5 个排队
- 实际并发还是 3 个（其他 5 个状态卡在 PENDING）

### Q29: 用户取消后子 Agent 还在跑怎么办？

**排查：**

1. **日志确认 cancel_event 设置**：
   - `request_cancel_background_task(task_id)` 是否被调用？
   - 看 `[trace=xxx] Requested cancellation for background task xxx`
2. **子 Agent 当前在哪个阶段**：
   - LLM 调用中：等当前调用返回（最多几十秒）
   - 工具调用中：等工具完成（可能 60+ 秒）
   - astream chunk 边界：立即检测到 → 退出

**最坏情况：** bash 工具跑 600 秒不返回 → cancel_event 检查不到 → 等到 timeout_seconds 触发（900 秒）才 TIMED_OUT

**用户体验：** 即使后台还在跑，前端 task_tool 协程已经 return（用户已经收到取消反馈）。延迟清理协程在后台异步收尾。

**根治方案：** 给重型工具加内部 timeout（例如 bash 工具内部 `subprocess.run(timeout=30)`）

### Q30: 同一个 thread_id 多次请求会冲突吗？

**短回答：** 视具体情况。

**详细：**

1. **顺序请求**（用户上一条对话完成才发下一条）：✅ 安全。LangGraph 用 thread_id 做 checkpoint，新请求基于上次 state 续跑。
2. **并发请求**（用户在前端连发两条）：⚠️ 取决于配置：
   - 有 checkpointer 持久化（如 PostgresSaver）：第二个请求会等第一个完成才能拿到 state lock
   - 无 checkpointer（in-memory）：可能发生竞态 —— 两个请求各自基于旧 state 跑，最后写入的覆盖

3. **多 worker 同 thread_id**：
   - 没有 sticky 路由 → 两个 worker 各自跑同一个 thread → 两份独立的 state
   - 有 sticky 路由 → 安全（同 thread 进同 worker）

**实践建议：**
- 启用持久化 checkpointer
- 加 sticky session 路由
- 前端加 debounce / 禁用提交按钮直到上次响应完成

## 终极 30 秒电梯总结

> "DeerFlow 的主子智能体通信用 **'工具调用即委派 (tool-call-as-delegation)'** 范式：LLM 调 `task` 工具 → `task_tool` 后台启动独立 LangGraph Agent → 5 秒轮询 + SSE 推送 → ToolMessage 回流。
>
> 三大工程亮点：① **持久 daemon event loop** 避免 httpx 连接池撕裂 ② **协作式取消** (`threading.Event` + chunk 边界) 让长工具调用能优雅退出 ③ **三层并发防护** (Prompt + Middleware + ThreadPool) 层层兜底防 LLM 任性。
>
> Token 走两条独立链路：**消息维度** (`_subagent_usage_cache` → `TokenUsageMiddleware` → AIMessage.usage_metadata) 给前端展示；**审计维度** (`SubagentTokenCollector.records` → `RunJournal` 按 caller 分桶) 给后端计费。
>
> 边界诚实：① 子 Agent 默认不能 fork 孙 Agent（写死 `subagent_enabled=False`）② 当前是单进程多线程，跨机扩展需要改 `_background_tasks` 用 Redis ③ 协作式取消最坏延迟 = 当前 chunk 跑完时间。"
