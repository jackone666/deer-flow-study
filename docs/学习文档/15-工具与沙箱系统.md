# 15 工具与沙箱系统 — 三层抽象 + MCP 动态加载 + Skill 策略过滤

> 面试口径：DeerFlow 的工具系统是**三层架构**：① **Sandbox 抽象层**（统一文件 / 命令 / 文件系统接口，三种实现 LocalSandbox / AioSandbox / Container）② **Tool 注册层**（builtin 工具 + 配置工具 + MCP 动态工具 + subagent 工具）③ **Skill 策略层**（按 SKILL.md 的 allowed-tools 字段过滤）。**子 Agent 拿到的工具列表 = 全集 → tool_groups 过滤 → subagent_enabled 过滤 → SkillPolicy 过滤 → disallowed_tools 过滤**，五道关卡。这章把这套体系拆开。

**本章课程目标：**

- 理解 DeerFlow 的 Sandbox 抽象（一个接口，三种实现）
- 掌握 `get_available_tools` 的 6 步组装流程
- 看清 MCP 工具的动态加载与 DeferredToolFilter 隐藏机制
- 吃透 Skill 系统的 SKILL.md 格式与 allowed-tools 策略
- 知道工具调用从 LLM 决定到沙箱执行的完整链路

**学习建议：** 这章理论 + 代码混搭，建议**先记结构后看细节**。"工具是什么" → "怎么注册" → "怎么过滤" → "怎么执行" → "沙箱怎么共享"，按这五步串起来，记忆点不会乱。

## 2、Sandbox 抽象层

### 2.1 Sandbox 抽象基类

```python
# sandbox/sandbox.py
class Sandbox(ABC):
    _id: str
    
    @abstractmethod
    def execute_command(self, command: str) -> str:
        """在沙箱内执行 bash 命令."""
    
    @abstractmethod
    def read_file(self, path: str) -> str:
        """读取沙箱内文件内容."""
    
    @abstractmethod
    def download_file(self, path: str) -> bytes:
        """下载沙箱内文件的二进制内容."""
    
    @abstractmethod
    def list_dir(self, path: str, max_depth=2) -> list[str]:
        """列出目录内容."""
    
    @abstractmethod
    def write_file(self, path: str, content: str) -> None:
        """写入文件."""
    
    @abstractmethod
    def str_replace(self, path: str, old_str: str, new_str: str) -> None:
        """文件内字符串替换."""
    
    # ... 其他方法
```

**关键设计：** 抽象基类定义了**所有工具会用到的"文件系统操作"**，具体实现可以是本地、远程、容器。

### 2.2 SandboxProvider — 工厂模式

```python
class SandboxProvider(ABC):
    @abstractmethod
    async def get_sandbox(self, sandbox_id: str | None = None) -> Sandbox: ...
    
    @abstractmethod
    async def shutdown(self): ...

# 全局单例
_provider: SandboxProvider | None = None

def get_sandbox_provider(**kwargs) -> SandboxProvider:
    """根据 config.yaml 决定用哪个 Provider."""
    global _provider
    if _provider is None:
        config = get_app_config()
        if config.sandbox.provider == "local":
            _provider = LocalSandboxProvider(...)
        elif config.sandbox.provider == "aio":
            _provider = AioSandboxProvider(...)
        elif config.sandbox.provider == "container":
            _provider = ContainerSandboxProvider(...)
    return _provider
```

### 2.3 三种实现对比

| 实现 | 隔离级别 | 性能 | 安全 | 适用场景 |
| --- | --- | --- | --- | --- |
| **LocalSandbox** | 进程内 + 路径前缀检查 | 极快 | ⚠️ 弱（虚拟前缀防穿越） | 开发 / 本地调试 |
| **AioSandbox** | 远程容器（云沙箱服务） | 中 | ✅ 强（容器隔离） | 生产部署 |
| **ContainerSandbox** | 本地 Docker / Apple Container | 中 | ✅ 强 | 自部署 + 离线 |

#### LocalSandbox（默认，本地开发）

```python
# sandbox/local/local_sandbox.py
class LocalSandbox(Sandbox):
    """本地文件系统沙箱."""
    
    def __init__(self, base_path: Path, virtual_prefix: str = "/mnt/user-data"):
        self.base_path = base_path  # 实际路径，如 /tmp/deerflow/sandbox/sb_xxx
        self.virtual_prefix = virtual_prefix  # LLM 看到的路径，如 /mnt/user-data
    
    def read_file(self, path: str) -> str:
        # 1. 路径穿越检查
        real_path = self._resolve_virtual_path(path)
        # 2. 实际读取
        return real_path.read_text()
    
    def _resolve_virtual_path(self, virtual: str) -> Path:
        """虚拟路径 → 实际路径，防穿越."""
        if not virtual.startswith(self.virtual_prefix):
            raise PermissionError(f"Path {virtual} not in {self.virtual_prefix}")
        relative = virtual[len(self.virtual_prefix):].lstrip("/")
        real = (self.base_path / relative).resolve()
        # 关键：resolve 后必须仍在 base_path 下（防 ../../etc/passwd）
        if not real.is_relative_to(self.base_path):
            raise PermissionError(f"Path traversal detected: {virtual}")
        return real
```

**关键防护：路径穿越检查**。

#### AioSandbox（生产推荐）

```python
class AioSandbox(Sandbox):
    """远程沙箱（通过 HTTP API 调用云沙箱服务）."""
    
    def __init__(self, sandbox_id: str, client: httpx.AsyncClient, base_url: str):
        self.sandbox_id = sandbox_id
        self.client = client
        self.base_url = base_url
    
    async def execute_command(self, command: str) -> str:
        response = await self.client.post(
            f"{self.base_url}/sandboxes/{self.sandbox_id}/exec",
            json={"command": command},
            timeout=300,
        )
        return response.json()["output"]
```

**好处：**
- 真容器隔离（即使代码注入也跑不出沙箱）
- 跨进程 / 跨机器（多 worker 共享同一个沙箱）
- 由 SaaS 厂商维护（沙箱镜像 / 安全更新不用自己管）

**代价：** 网络延迟（每次工具调用要 HTTP round-trip）

### 2.4 sandbox 在 state 里怎么共享

```python
# 主 Agent state
state["sandbox"] = {"sandbox_id": "sb_xxx"}

# task_tool 提取并传给子 Agent
sandbox_state = runtime.state.get("sandbox")  # 引用传递
SubagentExecutor(..., sandbox_state=sandbox_state)

# 子 Agent _build_initial_state
state["sandbox"] = self.sandbox_state  # 同一个 dict
```

**关键：所有同 thread 的 Agent 共享同一个 sandbox_id**，操作落在同一个文件系统上。

工具实现内部：

```python
# tools/builtins/bash_tool.py（伪代码）
@tool
async def bash(runtime: Runtime, command: str) -> str:
    sandbox_state = runtime.state.get("sandbox")
    if sandbox_state is None:
        raise RuntimeError("No sandbox available")
    
    provider = get_sandbox_provider()
    sandbox = await provider.get_sandbox(sandbox_state["sandbox_id"])
    return await sandbox.execute_command(command)
```

## 3、工具组装：`get_available_tools` 的 6 步流程

### 3.1 完整代码（精简）

```python
# tools/tools.py:46-150
def get_available_tools(
    groups: list[str] | None = None,
    include_mcp: bool = True,
    model_name: str | None = None,
    subagent_enabled: bool = False,
    *,
    app_config: AppConfig | None = None,
) -> list[BaseTool]:
    """组装当前可用工具列表."""
    config = app_config or get_app_config()
    
    # ── Step 1: 配置文件中的工具（按 group 过滤） ──
    tool_configs = [
        tool for tool in config.tools 
        if groups is None or tool.group in groups
    ]
    
    # ── Step 2: host bash 安全检查 ──
    if not is_host_bash_allowed(config):
        tool_configs = [t for t in tool_configs if not _is_host_bash_tool(t)]
    
    # ── Step 3: 实例化每个工具 ──
    loaded_tools = [
        _ensure_sync_invocable_tool(resolve_variable(cfg.use, BaseTool))
        for cfg in tool_configs
    ]
    
    # ── Step 4: 加 builtin 工具 ──
    builtin_tools = BUILTIN_TOOLS.copy()
    
    # 4a: skill_evolution 启用 → 加 skill_manage
    if getattr(config.skill_evolution, "enabled", False):
        from deerflow.tools.skill_manage_tool import skill_manage_tool
        builtin_tools.append(skill_manage_tool)
    
    # 4b: subagent_enabled → 加 task 工具
    if subagent_enabled:
        builtin_tools.extend(SUBAGENT_TOOLS)  # [task_tool]
    
    # 4c: 模型支持视觉 → 加 view_image
    model_config = config.get_model_config(model_name)
    if model_config and model_config.supports_vision:
        builtin_tools.append(view_image_tool)
    
    # ── Step 5: MCP 工具（动态加载的 MCP server 工具） ──
    if include_mcp:
        from deerflow.mcp.tools import get_mcp_tools_cached
        mcp_tools = get_mcp_tools_cached()  # 缓存（启动时初始化）
        loaded_tools.extend(mcp_tools)
    
    # ── Step 6: 合并 + 去重 + 排序 ──
    all_tools = loaded_tools + builtin_tools
    seen_names = set()
    deduped = []
    for tool in all_tools:
        if tool.name not in seen_names:
            seen_names.add(tool.name)
            deduped.append(tool)
    
    return sorted(deduped, key=lambda t: t.name)
```

### 3.2 关键要点

| 步骤 | 关键行为 |
| --- | --- |
| Step 1 | `tool_groups` 是用户配置的工具组（如只允许 "research" 组） |
| Step 2 | LocalSandbox 模式下 host bash 默认禁用（防代码注入） |
| Step 3 | `_ensure_sync_invocable_tool` 包裹 async-only 工具，让它们在 sync 上下文也能跑 |
| Step 4 | 4 个 builtin 是常驻的，按条件加可选项 |
| Step 5 | MCP 工具是**进程启动时一次性加载到缓存**，运行时直接读 |
| Step 6 | 同名去重 —— 配置工具优先于 builtin |

### 3.3 主子 Agent 工具列表差异

```python
# 主 Agent（task_tool 启用）
tools = get_available_tools(model_name="...", subagent_enabled=True)
# → [bash, ls, read_file, write_file, str_replace, web_search, ..., task]

# 子 Agent（task_tool 禁用，递归防护）
tools = get_available_tools(model_name="...", subagent_enabled=False)
# → [bash, ls, read_file, write_file, str_replace, web_search, ...]
#                                                                    ^^^
#                                                          没有 task 了
```

## 5、MCP 工具动态加载

### 5.1 MCP（Model Context Protocol）是什么

MCP 是 Anthropic 推出的协议（[modelcontextprotocol.io](https://modelcontextprotocol.io)），让外部工具服务能以**统一协议**接入 LLM Agent。

**好处：**
- 工具开发者只需实现 MCP server，不绑定具体 Agent 框架
- Agent 框架按 MCP 客户端规范连接，自动获取工具列表

### 5.2 MCP 在 DeerFlow 的位置

```python
# config.yaml
mcp:
  servers:
    - name: "github"
      command: "npx"
      args: ["@modelcontextprotocol/server-github"]
      env: {GITHUB_TOKEN: "..."}
    
    - name: "filesystem"
      command: "uvx"
      args: ["mcp-server-filesystem", "--root", "/data"]
```

**启动流程：**

```python
# 应用启动时（一次性）
async def initialize_mcp_tools():
    for server_config in app_config.mcp.servers:
        # 1. 启动 MCP server 进程（stdio 或 sse）
        client = MCPClient(server_config)
        await client.connect()
        
        # 2. 拉取工具列表
        tools = await client.list_tools()
        
        # 3. 包装成 LangChain BaseTool
        for tool_def in tools:
            wrapped = wrap_mcp_tool(client, tool_def)
            _MCP_TOOLS_CACHE.append(wrapped)
```

### 5.3 MCP Session Pool

```python
# mcp/session_pool.py
class MCPSessionPool:
    """MCP 客户端连接池."""
    
    def __init__(self):
        self._sessions: dict[str, MCPClient] = {}  # server_name → client
    
    async def get_or_create(self, server_name: str) -> MCPClient:
        if server_name not in self._sessions:
            self._sessions[server_name] = await self._create_client(server_name)
        return self._sessions[server_name]
```

**为什么要池化：**
- 每次工具调用都新建 MCP server 进程会很慢（启动 100-500ms）
- 复用连接 + 异步 IO 让多个工具调用并发

### 5.4 DeferredToolFilter（隐藏 MCP schema）

**问题：** 如果有 50 个 MCP 工具，全部展示给 LLM → system prompt 爆炸。

**DeerFlow 解法：**
1. **`tool_search` 元工具**：LLM 主动调用 `tool_search(query="github issue")` → 返回相关工具
2. **`DeferredToolFilterMiddleware`**：默认隐藏所有 MCP 工具的 schema，LLM 看不到
3. **promote 机制**：LLM 通过 `tool_search` "提升"某个工具到可见集合 → 后续轮次能直接调用

```python
# state["promoted"] = {"catalog_hash": "...", "names": ["github_create_issue", ...]}

class DeferredToolFilterMiddleware:
    def wrap_model_call(self, request, handler):
        promoted_names = state.get("promoted", {}).get("names", [])
        
        # 过滤工具列表：保留所有非 deferred + 已提升的 deferred
        visible_tools = [
            t for t in request.tools
            if t.name not in self.deferred_names or t.name in promoted_names
        ]
        request.tools = visible_tools
        return handler(request)
```

**好处：**
- 静态展示工具 < 10 个，prompt 紧凑
- LLM 按需 search + promote，灵活
- 已 promoted 的工具持续可用，不需要重复 search

### 6.4 Skill 加载链路

```python
# subagents/executor.py:367-393
async def _load_skills(self) -> list[Skill]:
    if self.config.skills is not None and len(self.config.skills) == 0:
        return []  # 空列表 = 禁用
    
    storage = await asyncio.to_thread(get_or_new_skill_storage, ...)
    all_skills = await asyncio.to_thread(storage.load_skills, enabled_only=True)
    
    # 按 config.skills 白名单过滤
    if self.config.skills is not None:
        allowed = set(self.config.skills)
        return [s for s in all_skills if s.name in allowed]
    
    return all_skills
```

**关键点：**
- 从磁盘加载所有启用的 SKILL.md
- 按 `config.skills` 白名单过滤（None=全部，[]=禁用，列表=按名）

### 6.6 allowed-tools 策略过滤（关键设计）

**SKILL.md 的 allowed-tools 字段决定加载该 skill 时只能用哪些工具：**

```python
# skills/tool_policy.py
def filter_tools_by_skill_allowed_tools(tools, skills):
    """按已启用 skill 的 allowed-tools 取交集过滤工具列表."""
    allowed = allowed_tool_names_for_skills(skills)
    if allowed is None:
        return tools  # 任意 skill 不限制 → 全部工具可用
    return [t for t in tools if t.name in allowed]


def allowed_tool_names_for_skills(skills):
    """计算所有已启用 skill 的 allowed-tools 取并集.
    
    规则：
    - 任意 skill 的 allowed_tools=None → 不限制（return None）
    - 所有 skill 都有 allowed_tools → 取并集
    """
    all_allowed = set()
    has_unrestricted = False
    
    for skill in skills:
        if skill.allowed_tools is None:
            has_unrestricted = True
            break
        all_allowed.update(skill.allowed_tools)
    
    if has_unrestricted:
        return None
    return all_allowed
```

**业务逻辑：**

```
case 1: 子 Agent 加载了 skill_a（allowed-tools=["read_file"]）+ skill_b（allowed-tools=["web_search"]）
        → 工具白名单 = {"read_file", "web_search"}
        → 只允许这两个工具

case 2: skill_a 不限制 + skill_b 限制
        → 整体不限制（最宽松者赢）

case 3: 没加载任何 skill
        → 不限制
```

**设计哲学：** Skill 是"我会用什么"的声明，多个 skill 累加得到"能用什么"。如果有任何 skill 说"全集"则全集。

## 7、五道关卡：子 Agent 拿到工具的完整链路

```
父 Agent 全部工具
    │
    ▼ 关卡 1：tool_groups 过滤（继承父限制）
父 group 限制后
    │
    ▼ 关卡 2：subagent_enabled=False（移除 task）
不含 task
    │
    ▼ 关卡 3：SubagentConfig.tools 白名单（如 bash 子 Agent 只允许 5 个）
config.tools 限制后
    │
    ▼ 关卡 4：SubagentConfig.disallowed_tools 黑名单（默认 [task, ask_clarification, present_files]）
黑名单过滤后
    │
    ▼ 关卡 5：SkillPolicy 过滤（按 SKILL.md allowed-tools 取并集）
最终子 Agent 工具列表
```

### 实例推演

**配置：**

```yaml
# 父 Agent
tool_groups: ["research", "code"]   # 限制 group

# 子 Agent (config.yaml.subagents.custom_agents.code_reviewer)
custom_agents:
  code_reviewer:
    tools: ["read_file", "ls", "bash"]   # 显式白名单
    disallowed_tools: ["task", "bash"]    # 显式黑名单（覆盖默认）
    skills: ["code_review_format"]
```

**code_review_format/SKILL.md：**
```yaml
---
allowed-tools: read_file
---
```

**链路推演：**

```
关卡 1: tool_groups=["research", "code"] → 父全集 - other_groups → {ls, bash, read_file, web_search, task}
关卡 2: subagent_enabled=False → {ls, bash, read_file, web_search}
关卡 3: config.tools=["read_file", "ls", "bash"] → 取交集 → {ls, bash, read_file}
关卡 4: disallowed_tools=["task", "bash"] → {ls, read_file}
关卡 5: skill code_review_format allowed-tools=["read_file"] → {read_file}

最终：子 Agent 只有 read_file 工具。
```

**这五道关卡分布在不同代码位置：**

| 关卡 | 代码位置 |
| --- | --- |
| 1 | `tools/tools.py:get_available_tools` 的 `groups` 参数 |
| 2 | 同上的 `subagent_enabled` 参数 |
| 3 | `subagents/executor.py:_filter_tools` 的 `allowed` 参数 |
| 4 | 同上的 `disallowed` 参数 |
| 5 | `subagents/executor.py:_apply_skill_allowed_tools` |

## 8、本章 ❓→💡 问答

### Q1：为什么 Sandbox 要做抽象，不能直接用 Python `subprocess`？

**A：** 三个理由：

1. **执行环境隔离**：直接用 subprocess 会在主机执行命令 —— LLM 注入 `rm -rf /` 就崩了。Sandbox 提供路径前缀检查、容器隔离等安全机制。
2. **跨实现切换**：开发用 LocalSandbox，生产换 AioSandbox，工具代码不用改（SOLID 的 D —— 依赖反转）。
3. **多线程共享**：父子 Agent 共享同一个 sandbox_id，操作落到同一个文件系统 —— 不抽象的话每次新建 subprocess 各自独立。

### Q2：MCP 工具为什么要 DeferredFilter 隐藏？直接全部展示不行吗？

**A：** Token 成本。

每个工具的 schema（name + description + parameters）平均 200-500 token。50 个 MCP 工具 = 10k-25k token，每次 LLM 调用都消耗。

DeferredFilter 把所有 MCP 工具默认隐藏，只暴露 `tool_search` 元工具。LLM 主动 search → 提升所需工具到可见集合 → 后续轮次直接用。

**节省：** 启动只暴露 ~10 个静态工具（~3k token），按需 promote 4-5 个 MCP 工具（~1.5k token）—— 总 token 成本降到 1/5。

### Q3：Skill 的 allowed-tools 取**并集**而不是**交集**，安全吗？

**A：** 业务上合理。

Skill 是"我会做这个事，需要这些工具"。多个 skill 累加 = 多种能力组合。

```
skill_a: 写代码（allowed-tools=[read, write, bash]）
skill_b: 写文档（allowed-tools=[read, write_file]）

并集：{read, write, write_file, bash}
```

这是合理的：加载这两个 skill 的子 Agent 既能写代码也能写文档。

**安全防线在 disallowed_tools**：如果想强制禁用某工具（如 bash），在 SubagentConfig.disallowed_tools 里加，不依赖 skill。

### Q4：MCP server 启动慢（500ms）怎么办？

**A：** DeerFlow 已经做了优化：

1. **进程启动时一次性初始化**：`initialize_mcp_tools()` 在 FastAPI startup 钩子里调用，不在请求路径上
2. **Session Pool 复用连接**：避免每次工具调用都新建 server 进程
3. **缓存工具列表**：`get_mcp_tools_cached()` 直接读内存，不重新拉

**真正慢的场景：** MCP server 第一次响应（冷启动），后续调用都是毫秒级。

### Q5：bash 工具被禁用时（host bash 不允许），子 Agent 还能跑命令吗？

**A：** 看子 Agent 配置：

- 如果是 `BASH_AGENT_CONFIG`（tools=["bash", ...]）→ host bash 禁用时**bash 子 Agent 整体不可用**（task_tool 拒绝该 subagent_type）
- 如果是 general-purpose 子 Agent → bash 不在工具列表 → LLM 看不到 → 不会调用

**不允许 host bash 的两种用法：**
1. AioSandbox 模式：bash 通过沙箱 API 跑（不是 host）—— 仍可用
2. 完全禁 bash：所有 shell 操作都不行 → 影响功能

```python
# task_tool.py:371-374
if subagent_type == "bash":
    host_bash_allowed = is_host_bash_allowed(...)
    if not host_bash_allowed:
        return f"Error: {LOCAL_BASH_SUBAGENT_DISABLED_MESSAGE}"
```

## 9、本章总结

**工具系统三层抽象：**

| 层 | 关键文件 | 职责 |
| --- | --- | --- |
| 注册层 | `tools/tools.py` | get_available_tools 6 步组装 |
| 工具实现层 | `tools/builtins/*.py` | 具体工具函数 |
| 沙箱层 | `sandbox/sandbox.py` + 三种实现 | 文件 / 命令统一接口 |

**子 Agent 工具五道关卡：**

```
全集 → tool_groups → subagent_enabled → config.tools → disallowed_tools → skill allowed-tools
```

**Skill 系统三个特征：**
- Markdown 文件存储（人类可读可编辑）
- 加载到 system prompt（不是检索）
- allowed-tools 控制工具策略（取并集）

**MCP 三个机制：**
- 启动期初始化（一次性）+ Session Pool 复用
- DeferredFilter 默认隐藏（节省 token）
- tool_search + promote 按需提升

**面试金句：**

> "DeerFlow 工具系统的设计哲学是 **'最小权限 + 多重过滤 + 按需提升'**：
> - 子 Agent 默认拿不到 task 工具（递归防护）
> - SubagentConfig 提供 tools 白名单 + disallowed_tools 黑名单
> - Skill 系统通过 allowed-tools 字段进一步收紧
> - MCP 工具默认隐藏，让 LLM 用 tool_search 元工具按需提升
> - 沙箱抽象层让工具代码不绑定具体执行环境（local / aio / container）
>
> 这套设计在 **token 成本**（隐藏不必要的工具 schema）和 **安全**（多重过滤防止子 Agent 越权）之间取得平衡。"
