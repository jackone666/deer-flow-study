# 08 协作式取消 — threading.Event + asyncio.shield + 延迟清理

> 面试口径：DeerFlow 的子 Agent 取消机制是 **"协作式 (cooperative)"** 而不是"抢占式 (preemptive)"。`threading.Event` 在 `astream` 的每个 chunk 边界被检查，**子 Agent 自己判断"该退了"**，不能强杀。原因：① `Future.cancel()` 杀不死正在跑的 sync 工具调用 ② `asyncio.CancelledError` 会撕裂 httpx 连接池 ③ 子 Agent 需要走完 try/finally 清理资源。看完这章你能讲清楚"用户在前端按取消，到子 Agent 真正退出，中间发生了什么"。

**本章课程目标：**

- 理解为什么 DeerFlow 用 `threading.Event` 而不是 `Future.cancel()` / `task.cancel()`
- 吃透取消的三种触发路径：① 主 Agent 协程被取消 ② 线程池超时 ③ 轮询超时（兜底）
- 掌握 `asyncio.shield` 在异常处理路径里的作用 —— 给 token 上报留一个救命窗口
- 知道延迟清理协程 (`_deferred_cleanup_subagent_task`) 解决的边缘问题

**学习建议：** 这章是**生产环境真问题**章节。建议读完后想象 3 个故障场景：① 用户取消时子 Agent 卡在长 LLM 调用 ② 子 Agent 内部抛异常时 cancel_event 已经被 set ③ 父协程被取消时还想抢救 token 数据。每个场景都用本章代码模拟一遍。

## 1、本章导读

**取消的核心问题：怎么"礼貌地"打断一个正在跑的 LLM Agent？**

子 Agent 在跑的时候，状态可能是：
1. 在 `astream` 的 yield 边界（chunk 之间）—— 容易打断
2. 在 LLM 调用中（HTTP 请求等响应）—— 难打断
3. 在工具调用中（比如 bash 执行 60 秒）—— 几乎不能打断
4. 在 await 某个 IO（读文件 / 等数据库）—— 容易打断

`asyncio.CancelledError` 能打断的是 **1 和 4**，**2 和 3 打不断**。所以"硬取消"不可行。

DeerFlow 的方案：**协作式取消** —— 子 Agent 在每个 chunk 边界主动检查 `cancel_event`，发现被设置了就主动退出。代价是延迟（最坏要等当前 chunk 跑完，可能几秒到几十秒），收益是干净退出 + 资源不泄漏。

## 2、三种取消触发路径

### 2.1 路径 A：用户在前端点取消

```
前端 → /cancel API → request_cancel_background_task(task_id)
        ↓
        result.cancel_event.set()
        ↓
        子 Agent astream 下一个 chunk 边界检查到 → 退出
        ↓
        try_set_terminal(CANCELLED)
        ↓
        主 Agent 5 秒轮询命中 CANCELLED → 返回 "Task cancelled by user."
```

### 2.2 路径 B：线程池超时

```
SubagentExecutor.execute_async 的 run_task 内部：
    execution_future = _submit_to_isolated_loop_in_context(...)
    try:
        execution_future.result(timeout=self.config.timeout_seconds)  # ← 阻塞等
    except FuturesTimeoutError:                                       # ← 900 秒到
        result_holder.cancel_event.set()                              # ← 通知子 Agent 退
        result_holder.try_set_terminal(SubagentStatus.TIMED_OUT, ...)
        execution_future.cancel()                                      # ← 同时也尝试 cancel future
```

`execution_future.cancel()` 在 `concurrent.futures.Future` 上**通常不生效**（任务已经在跑），主要靠 `cancel_event.set()` 触发协作式退出。

### 2.3 路径 C：轮询超时（兜底）

```python
# task_tool.py:562-574
if poll_count > max_poll_count:
    timeout_minutes = config.timeout_seconds // 60
    logger.error(f"...polling timed out after {poll_count} polls...")
    _report_subagent_usage(runtime, result)
    ...
    request_cancel_background_task(task_id)
    _schedule_deferred_subagent_cleanup(task_id, trace_id, max_poll_count)
    return f"Task polling timed out after {timeout_minutes} minutes..."
```

**何时触发？** `max_poll_count = (config.timeout_seconds + 60) // 5`。
- 默认 timeout_seconds=900 → max_poll_count=192 次轮询
- 192 × 5 = 960 秒 = 16 分钟

**为什么有这层兜底？**

理论上路径 B（线程池超时）应该先触发：
- timeout_seconds=900 秒触发线程池超时
- cancel_event.set + try_set_terminal(TIMED_OUT)
- 主 Agent 第 181 次轮询命中 TIMED_OUT，正常返回

但如果 **后台调度线程死锁 / 卡死**（极端情况），线程池的 `Future.result(timeout=)` 可能不返回。这时 max_poll_count 兜底 —— 主 Agent 协程不能永远阻塞。

### 2.4 路径 D：主 Agent 协程被取消（CancelledError）

`task_tool` 的协程被外层取消（比如 LangGraph 因为某个其他错误中断整个 agent run）：

```python
# task_tool.py:576-604
except asyncio.CancelledError:
    request_cancel_background_task(task_id)
    
    terminal_result = None
    try:
        terminal_result = await asyncio.shield(_await_subagent_terminal(task_id, max_poll_count))
    except asyncio.CancelledError:
        pass
    
    final_result = terminal_result or get_background_task_result(task_id)
    if final_result is not None:
        _report_subagent_usage(runtime, final_result)
    
    if final_result is not None and _is_subagent_terminal(final_result):
        cleanup_background_task(task_id)
    else:
        _schedule_deferred_subagent_cleanup(task_id, trace_id, max_poll_count)
    
    _subagent_usage_cache.pop(tool_call_id, None)
    raise  # 重抛
```

这是**最复杂的路径**，下一节专门讲。

## 3、CancelledError 路径深度解析

### 3.1 为什么需要 `asyncio.shield`

`asyncio.shield` 的作用：**让被包裹的协程不受外层 cancel 影响**。

```python
terminal_result = None
try:
    terminal_result = await asyncio.shield(_await_subagent_terminal(task_id, max_poll_count))
except asyncio.CancelledError:
    pass
```

**没有 shield 会怎样？**

```python
# 错误示范
terminal_result = await _await_subagent_terminal(task_id, max_poll_count)
# 外层 cancel 立即穿透 → _await 内部的 await asyncio.sleep(5) 被中断 → CancelledError 抛出
# 但子 Agent 还在跑，token 数据没收集完
```

**用 shield 怎么改善？**

```python
# 正确做法
terminal_result = await asyncio.shield(_await_subagent_terminal(...))
# 外层 cancel 不会穿透到 _await 内部
# _await 能正常跑完 → 拿到子 Agent 终止状态 → token 数据收集完整
```

### 3.2 但 shield 不是万能的

```python
try:
    terminal_result = await asyncio.shield(_await_subagent_terminal(...))
except asyncio.CancelledError:
    pass  # ← shield 也可能被穿透
```

**什么情况下 shield 被穿透？**

`asyncio.shield(coro)` 的语义：
- shield 创建一个 Task wrapper
- 外层 cancel 时，shield **本身会抛出 CancelledError**（不传给内部）
- 内部 task 继续跑（直到自然完成）

但如果 `await shield(...)` 这条语句抛 `CancelledError` 后，**外层又来一次 cancel**（嵌套取消）—— 第二次 cancel 会真的传到 shield 内部。

DeerFlow 这里 `try/except` 兜底：shield 抛出 CancelledError 时跳过等待，进入下一步（直接拿当前 result）。

### 3.3 异常路径的完整逻辑

```python
except asyncio.CancelledError:
    # ① 通知子 Agent 退出
    request_cancel_background_task(task_id)
    
    # ② 用 shield 给子 Agent 一个机会跑完
    terminal_result = None
    try:
        terminal_result = await asyncio.shield(_await_subagent_terminal(...))
    except asyncio.CancelledError:
        pass  # shield 也被穿透了，放弃等
    
    # ③ 上报最终 token（即使没等到终止也要上报已收集的）
    final_result = terminal_result or get_background_task_result(task_id)
    if final_result is not None:
        _report_subagent_usage(runtime, final_result)
    
    # ④ 清理：已终止立即清理；否则调度延迟清理（不阻塞）
    if final_result is not None and _is_subagent_terminal(final_result):
        cleanup_background_task(task_id)
    else:
        _schedule_deferred_subagent_cleanup(task_id, trace_id, max_poll_count)
    
    # ⑤ 清理 cache（防止泄漏）
    _subagent_usage_cache.pop(tool_call_id, None)
    
    # ⑥ 必须重抛 —— LangGraph 框架要靠它知道协程被取消
    raise
```

**4 个核心动作：**

| 动作 | 目的 | 代码 |
| --- | --- | --- |
| 通知 | 让子 Agent 知道要退 | `request_cancel_background_task` → `cancel_event.set()` |
| 等待 | 给子 Agent 收集 token 的机会 | `asyncio.shield(_await_subagent_terminal)` |
| 上报 | 抢救 token 数据 | `_report_subagent_usage` |
| 清理 | 防止资源泄漏 | `cleanup` 或 `_schedule_deferred_cleanup` |

## 4、threading.Event 在 astream 边界检查

### 4.1 子 Agent 端的检查代码

```python
# executor.py:540-552
async for chunk in agent.astream(state, config=run_config, context=context, stream_mode="values"):
    # ── 协作式取消检查 ──
    if result.cancel_event.is_set():
        logger.info(f"Subagent {self.config.name} cancelled by parent")
        result.try_set_terminal(
            SubagentStatus.CANCELLED,
            error="Cancelled by user",
            token_usage_records=collector.snapshot_records(),
        )
        return result
    
    final_state = chunk
    ...
```

### 4.2 为什么 chunk 边界？

`agent.astream(stream_mode="values")` 在每次 state 更新时 yield 一个 chunk。所以"chunk 边界"对应：
- LLM 调用完成
- 工具调用完成
- 中间件更新 state 完成

这些都是**自然的检查点**，因为：
- LLM 调用中检查没用（HTTP 还在传输）
- 工具调用中检查没用（同步代码不可中断）
- 只有"一阶段完成 → 即将进入下阶段"的边界能优雅退出

### 4.3 启动前预检

```python
# executor.py:531-538
# 启动前预检：如果已经被取消就直接返回
if result.cancel_event.is_set():
    logger.info(f"Subagent {self.config.name} cancelled before streaming")
    result.try_set_terminal(
        SubagentStatus.CANCELLED,
        error="Cancelled by user",
        token_usage_records=collector.snapshot_records(),
    )
    return result
```

**为什么需要预检？**

竞态场景：
- T0：`task_tool.execute_async` 提交任务到 ThreadPool
- T1：用户立即点取消 → `cancel_event.set()`
- T2：调度线程开始跑 `_aexecute`，但 astream 还没开始

如果不预检，`astream` 会先发起一次 LLM 调用，浪费 token —— 预检在第一次 LLM 调用之前就退出。

## 5、延迟清理协程

### 5.1 问题场景

主 Agent 轮询超时（路径 C），调用 `request_cancel_background_task` 后立即 return。但**子 Agent 实际上还在跑**（卡在某个工具调用里），cancel_event 要等下一个 chunk 才生效。

如果 `task_tool` 立即 `cleanup_background_task(task_id)`：
- `_background_tasks[task_id]` 被删除
- 子 Agent 终止时调用 `try_set_terminal`，但 `result_holder` 还在（dataclass 实例）
- `_background_tasks` 不再有这个 entry，下次轮询查不到（其实这里已经 return 不轮询了，但其他模块可能查）

### 5.2 解决方案

```python
# task_tool.py:132-156
async def _deferred_cleanup_subagent_task(task_id, trace_id, max_polls):
    """对已取消的子 Agent 持续轮询，直到可以安全清理。"""
    cleanup_poll_count = 0
    while True:
        result = get_background_task_result(task_id)
        if result is None:
            return
        if _is_subagent_terminal(result):
            cleanup_background_task(task_id)
            return
        if cleanup_poll_count >= max_polls:
            logger.warning(f"Deferred cleanup for task {task_id} timed out after {cleanup_poll_count} polls")
            return  # 兜底超时
        await asyncio.sleep(5)
        cleanup_poll_count += 1
```

```python
# task_tool.py:173-186
def _schedule_deferred_subagent_cleanup(task_id, trace_id, max_polls):
    cleanup_task = asyncio.create_task(_deferred_cleanup_subagent_task(...))
    cleanup_task.add_done_callback(lambda task: _log_cleanup_failure(task, ...))
```

### 5.3 关键设计

| 设计点 | 说明 |
| --- | --- |
| `asyncio.create_task` 启动独立任务 | 不会被当前协程的 cancel 影响 |
| `add_done_callback` 处理失败日志 | 延迟清理协程异常不影响主流程 |
| 兜底 `max_polls` 超时 | 防止子 Agent 永远卡死导致清理永远不发生 |

### 5.4 fire-and-forget 模式

```python
# 不 await，立即返回
_schedule_deferred_subagent_cleanup(task_id, trace_id, max_poll_count)
return f"Task polling timed out..."
```

**为什么这样写？**

主 `task_tool` 协程已经决定要 return 了（用户层面任务完成）。延迟清理是**后台善后**工作，不能阻塞主返回。

`asyncio.create_task` 在 event loop 里独立运行 —— 即使父协程已 return，子任务继续跑直到完成。`add_done_callback` 处理任何异常防止 silently fail。

## 6、`Future.cancel()` 的局限性

### 6.1 `concurrent.futures.Future.cancel()` 行为

```python
# execute_async 内部
execution_future = _submit_to_isolated_loop_in_context(...)
try:
    execution_future.result(timeout=self.config.timeout_seconds)
except FuturesTimeoutError:
    result_holder.cancel_event.set()    # 主要靠这个
    result_holder.try_set_terminal(SubagentStatus.TIMED_OUT, ...)
    execution_future.cancel()            # 辅助调用
```

**官方文档：**

> If the call is currently being executed or finished running and cannot be cancelled then the method will return False, otherwise the call will be cancelled and the method will return True.

翻译：**如果任务已经在跑，cancel 返回 False，不能真的取消**。

`run_task` 已经在调度线程池跑 → `_submit_to_isolated_loop_in_context` 已经把协程提交到 daemon loop → `_aexecute` 已经在跑。这个时候 `Future.cancel` 啥都做不了。

### 6.2 为什么还要调

```python
execution_future.cancel()  # 反正调一下也没坏处
```

两个理由：
1. **防御性**：万一未来某个版本 cancel 能更激进（比如调用 `loop.cancel(coro)`），保留这行
2. **状态标记**：`Future.cancelled() == True` 让其他代码知道这个 Future 已经放弃

### 6.3 真正起作用的是 cancel_event

```python
result_holder.cancel_event.set()  # ← 这才是关键
```

子 Agent 在 daemon loop 里跑，下一个 chunk 边界检查 `cancel_event.is_set() == True` → 主动退出。

## 7、本章 ❓→💡 问答

### Q1：为什么不用 `asyncio.Task.cancel()` 直接取消子 Agent 协程？

**A：** 跨事件循环 + 跨线程的 cancel 非常脆弱：

- 子 Agent 协程在 daemon loop 里
- 主协程在主 loop 里
- 主协程拿到的是 `concurrent.futures.Future`（不是 `asyncio.Task`），没有 `cancel` 等价方法

如果硬要用 `asyncio.Task.cancel`：
- 要把 daemon loop 里的 task 引用传出来
- 跨 loop 调用 cancel 要用 `loop.call_soon_threadsafe(task.cancel)` —— 复杂且容易竞态
- 即使 cancel 成功，`CancelledError` 抛出时可能撕裂 httpx 等绑定 loop 的资源

`threading.Event` 是**最简单可靠的跨线程信号**，单独配合 chunk 边界检查就够了。

### Q2：如果子 Agent 卡在一个 60 秒的 bash 工具调用，cancel_event 多久生效？

**A：** 至少 60 秒（直到当前工具调用完成 + 下一个 chunk 边界检查）。这是协作式取消的代价。

DeerFlow 没有更激进的中断机制 —— 因为：
- bash 执行可能是真的关键操作（数据库迁移），强杀会损坏数据
- 工具实现可以自己加超时（`subprocess.run(timeout=30)`）
- 用户体验上 60 秒等待 vs 资源损坏，前者可接受

### Q3：`asyncio.shield` 内部的 `_await_subagent_terminal` 完成后，shield 还有效吗？

**A：** shield 的语义是"包裹一个 awaitable，外层 cancel 时只 cancel 包装层不传给内部"。

```python
async def _await_subagent_terminal(...):
    for _ in range(max_polls):
        result = get_background_task_result(task_id)
        if _is_subagent_terminal(result):
            return result   # ← 正常返回
        await asyncio.sleep(5)
    return None
```

如果 `_await_subagent_terminal` 正常返回了 → shield 包装的 task 完成 → `await shield(...)` 拿到返回值 —— 不再有"shield 中"这个状态。

如果 `await shield(...)` 等的过程中外层 cancel：
- shield 抛 CancelledError 给外层
- shield 内部的 task 继续在后台跑（直到完成或永远跑）
- DeerFlow 这里没存 task 引用，跑完结果丢弃 —— 但 token 数据已经在 SubagentResult 里，由其他路径处理

### Q4：延迟清理协程如果在主进程退出时还没跑完会怎样？

**A：** `asyncio.create_task` 创建的 task 是**普通 task**（不是 daemon），但 daemon loop 是 daemon 线程。

主进程退出时：
- 主线程的 event loop 退出 → 它的 `asyncio.create_task` 创建的 task 被丢弃
- daemon 线程被强制退出 → daemon loop 上的所有协程被丢弃

但**这个 cleanup_task 不在 daemon loop 上，是在主 loop 上**！主 loop 退出时它会被丢弃。

实际影响：
- `_background_tasks[task_id]` 这个 entry 永远不删 —— 但 Python 进程都退出了，dict 也跟着销毁，没真正泄漏
- 子 Agent 内部的资源（httpx 连接 / 文件句柄）会被 daemon loop 关闭时统一释放

所以 **进程退出时延迟清理失败不会导致真正泄漏**，只是在 `atexit` 里看不到清理日志。

### Q5：如果 `cancel_event.set()` 调了但子 Agent 内部还要再 LLM 调用一次怎么办？

**A：** 会浪费一次 LLM 调用 —— astream yield 出 chunk 之前不会检查。

时序：
```
T0: cancel_event.set()
T1: 子 Agent 在 LLM 调用中（30 秒）
T2: LLM 返回 → astream 准备 yield → 中间件链 → 工具调用 → ...
T3: 下一个 yield → 检查 cancel_event → 退出
```

T1-T3 之间的 token 都浪费了。**这是协作式取消的固有成本**。

如果非常在意，可以在 LLM 调用层面加 timeout（比如 httpx 的请求级超时），但会引入新问题（重试 / 局部失败处理），DeerFlow 选了简单方案。

## 8、本章总结

**取消的四条路径：**

| 路径 | 触发者 | 延迟 |
| --- | --- | --- |
| A 用户取消 | 前端 → request_cancel_background_task | 最坏 1 个 chunk（几秒） |
| B 线程池超时 | timeout_seconds | 准时（默认 900 秒）+ 1 个 chunk |
| C 轮询超时 | max_poll_count | 准时（约 960 秒）+ 1 个 chunk |
| D 主协程取消 | LangGraph 上层 cancel | 立即返回（不等子 Agent） |

**核心机制：**

```
threading.Event (跨线程信号)
        ↓
astream chunk 边界 (检查时机)
        ↓
return result (优雅退出)
        ↓
try_set_terminal(CANCELLED) (状态切换)
        ↓
collector.snapshot_records() (token 抢救)
```

**异常路径关键 API：**

```
asyncio.shield(coro)        → 防止外层 cancel 穿透
asyncio.create_task(coro)    → fire-and-forget 后台清理
add_done_callback             → 处理失败不阻塞主流程
```

**记忆口诀：**
> Event 通知不强杀，边界检查求优雅；shield 包裹保 token，延迟清理防泄漏。
