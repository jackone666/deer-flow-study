# 09 三层并发防护 — Prompt 软约束 / Middleware 硬截断 / ThreadPool 物理上限

> 面试口径：DeerFlow 限制并发子 Agent 用了 **三道防线**：① **System Prompt 软约束**（告诉 LLM "每次最多 N 个 task"，但 LLM 可能不听话）② **`SubagentLimitMiddleware` 硬截断**（after_model 钩子直接改 tool_calls 列表）③ **`ThreadPoolExecutor(max_workers=3)` 物理上限**（即使中间件失效，调度池也只能跑 3 个）。**三层不是冗余，是层层兜底** —— 每层都解决前一层失败时的暴露面。

**本章课程目标：**

- 理解为什么"只用一层"不够：Prompt / Middleware / ThreadPool 各自的失效场景
- 吃透 `SubagentLimitMiddleware` 的截断算法（保留前 N 个 task，非 task 不计数）
- 知道 `_clamp_subagent_limit([2, 4])` 钳制范围背后的工程考量
- 掌握"为什么不允许 1 个或 5 个" 的设计哲学

**学习建议：** 这章重点是**层层防护**思想。建议读完后想象三个失败场景：① LLM 完全不读 prompt（旧模型 / 微调跑偏） ② 用户配置 `max_concurrent=10`（绕过中间件） ③ Middleware 抛异常没截断（补丁版本 bug）。三个场景下 DeerFlow 仍然不会"炸"—— 这就是分层防护的价值。

## 1、本章导读

并发限制的目标：**限制单次 LLM 响应中并发跑的子 Agent 数量上限到 [2, 4] 范围内**。

为什么要限制？
- **下游 LLM API 限流**：5+ 个并发 LLM 调用容易撞 RPS / TPM 上限
- **内存压力**：每个子 Agent 各自有 messages 历史 / 工具列表 / state，5+ 个并发吃内存
- **观察成本**：前端展示 5+ 个 task_running 事件流混乱
- **token 消耗**：并发 fork 容易导致单次响应消耗暴涨（5 个子 Agent × 100k context = 500k 输入 token）

DeerFlow 的解法：**三层兜底**。

```
LLM 输出 5 个 task 调用
        ↓
[Layer 1] System Prompt：告诉 LLM 别这么干
        │  ⚠️ 可失效原因：LLM 不听话 / 旧模型理解不了 / 用户改了 prompt
        ↓
[Layer 2] SubagentLimitMiddleware.after_model：硬截断到 3 个
        │  ⚠️ 可失效原因：subagent_enabled=False / Middleware 抛异常 / 配置错误
        ↓
[Layer 3] ThreadPoolExecutor(max_workers=3)：调度池物理上限
        │  ⚠️ 可失效原因：源码改了 max_workers / 启动多 Process（DeerFlow 不支持）
        ↓
真实并发上限永远 ≤ 3
```

## 2、Layer 1：System Prompt 软约束

### 2.1 prompt 内容

`agents/lead_agent/prompt.py:281-429` 中的 `_build_subagent_section`，注入到主 Agent 的 system prompt：

```
**🚀 子代理模式已激活——拆解、委派、综合**

**⛔ 硬性并发限制：每次响应最多 3 个 task 调用，这不是可选项。**
- 每次响应中，你最多只能包含 3 个 task 工具调用。多余的调用会被系统**静默丢弃**——你将失去这些工作。
- **启动子代理之前，你必须在思考中数出子任务数量：**
  - 如果 ≤ 3：本轮全部启动。
  - 如果 > 3：**本轮选择最重要的 3 个子任务。** 剩下的留到下轮。
- **多批次执行**（>3 个子任务时）：
  - 第 1 轮：并行启动子任务 1-3 → 等待结果
  - 第 2 轮：并行启动下一批 → 等待结果
  - ……继续直到所有子任务完成
  - 最后一轮：综合所有结果形成连贯答案
```

### 2.2 三个关键设计

| 设计点 | 目的 |
| --- | --- |
| 用 ⛔ 强语气 | 强模型（Claude / GPT-4）会重视显式约束 |
| 列出"被丢弃"后果 | 让 LLM 知道不遵守会浪费 token，自我约束 |
| 多批次执行示例 | 给 LLM 明确的 fallback 路径（不是绝望式截断） |
| 数字 `{n}` 模板化 | 通过 `_build_subagent_section(max_concurrent=3)` 注入实际值 |

### 2.3 为什么 prompt 不可靠

**典型失效场景：**

1. **旧模型 / 弱模型**：如 GPT-3.5 / Llama-7B，对长 prompt 中的指令服从度低
2. **温度过高**：temperature=1.5 时 LLM 输出更随机，可能忽略指令
3. **任务紧迫感强**：如 "迅速对比 10 家云厂商" → LLM 倾向一次性 fork 10 个
4. **用户改 prompt**：自定义 system prompt 时可能漏掉这一段

prompt 是"沟通"，**不是"强制"**。模型可以选择不听 —— 所以下层防护必须存在。

## 3、Layer 2：SubagentLimitMiddleware 硬截断

### 3.1 完整代码

```python
# agents/middlewares/subagent_limit_middleware.py
class SubagentLimitMiddleware(AgentMiddleware[AgentState]):
    def __init__(self, max_concurrent: int = MAX_CONCURRENT_SUBAGENTS):
        super().__init__()
        self.max_concurrent = _clamp_subagent_limit(max_concurrent)
    
    def _truncate_task_calls(self, state: AgentState) -> dict | None:
        messages = state.get("messages", [])
        if not messages:
            return None
        
        last_msg = messages[-1]
        if getattr(last_msg, "type", None) != "ai":
            return None
        
        tool_calls = getattr(last_msg, "tool_calls", None)
        if not tool_calls:
            return None
        
        task_indices = [i for i, tc in enumerate(tool_calls) if tc.get("name") == "task"]
        if len(task_indices) <= self.max_concurrent:
            return None  # 没超限，不动
        
        indices_to_drop = set(task_indices[self.max_concurrent:])
        truncated = [tc for i, tc in enumerate(tool_calls) if i not in indices_to_drop]
        
        dropped_count = len(indices_to_drop)
        logger.warning(f"Truncated {dropped_count} excess task tool call(s) from model response (limit: {self.max_concurrent})")
        
        updated_msg = clone_ai_message_with_tool_calls(last_msg, truncated)
        return {"messages": [updated_msg]}
    
    @override
    def after_model(self, state, runtime):
        return self._truncate_task_calls(state)
    
    @override
    async def aafter_model(self, state, runtime):
        return self._truncate_task_calls(state)
```

### 3.2 算法步骤

```
输入：state.messages 末条 AIMessage(tool_calls=[t1, t2, t3, t4, t5, t6])
       其中 t1-t5 是 task，t6 是 read_file

Step 1: 找到所有 task 调用的索引
       task_indices = [0, 1, 2, 3, 4]
       len = 5 > 3 (max_concurrent) → 需要截断

Step 2: 决定丢弃哪些
       保留前 3 个 task：indices [0, 1, 2]
       丢弃后 2 个 task：indices [3, 4]
       indices_to_drop = {3, 4}

Step 3: 重建 tool_calls 列表
       truncated = [t1, t2, t3, t6]   ← 注意 t6（非 task）保留
       注意：原顺序保持

Step 4: 替换最后一条 AIMessage
       clone 一个 AIMessage（同 id）→ LangGraph 视为替换
       return {"messages": [updated_msg]}
```

### 3.3 三个关键细节

#### 细节 1：只统计 task

```python
task_indices = [i for i, tc in enumerate(tool_calls) if tc.get("name") == "task"]
```

非 task 工具（read_file / web_search / bash 等）**不参与计数**。所以一次响应里可以：
- 3 个 task + 5 个 read_file + 2 个 web_search → 全部保留
- 5 个 task + 2 个 read_file → 截断到 3 个 task + 2 个 read_file

设计哲学：**限制的是"重型操作"**，普通工具是基础能力，不该限制。

#### 细节 2：保留前 N 个

```python
indices_to_drop = set(task_indices[self.max_concurrent:])
truncated = [tc for i, tc in enumerate(tool_calls) if i not in indices_to_drop]
```

保留 `task_indices[:max_concurrent]` —— **LLM 输出的前 max_concurrent 个 task**。

为什么不随机？为什么不"打分排序"？

| 方案 | 优点 | 缺点 |
| --- | --- | --- |
| 保留前 N 个 | 实现简单，LLM 通常先输出最重要的 | 如果 LLM 习惯把简单任务先写则出错 |
| 随机 N 个 | 公平 | 不可重现，难调试 |
| 打分排序 | 智能 | 引入打分模型，又是一次 LLM 调用 |

DeerFlow 选了"前 N 个" —— 配合 prompt 引导 LLM **"选择最重要的 3 个"**，让模型自我排序。

#### 细节 3：clone AIMessage 而不是修改

```python
updated_msg = clone_ai_message_with_tool_calls(last_msg, truncated)
return {"messages": [updated_msg]}
```

为什么不直接 `last_msg.tool_calls = truncated`？

- AIMessage 是 Pydantic 对象，可能 immutable
- LangGraph 用 message id 判断"替换还是新增"：clone 时保留 id → 替换原消息

如果不 clone：
- 修改原对象可能不被 LangGraph state 检测到（state 用 dict 比较）
- 多个 middleware 同时修改可能竞态

## 4、`_clamp_subagent_limit` 钳制 [2, 4]

### 4.1 实现

```python
MIN_SUBAGENT_LIMIT = 2
MAX_SUBAGENT_LIMIT = 4

def _clamp_subagent_limit(value: int) -> int:
    return max(MIN_SUBAGENT_LIMIT, min(MAX_SUBAGENT_LIMIT, value))
```

### 4.2 用户配置

```python
# agent.py 主 Agent 构建
subagent_enabled = cfg.get("subagent_enabled", False)
if subagent_enabled:
    max_concurrent_subagents = cfg.get("max_concurrent_subagents", 3)
    middlewares.append(SubagentLimitMiddleware(max_concurrent=max_concurrent_subagents))
```

用户在 RunnableConfig 里传 `max_concurrent_subagents=10`，会被 clamp 到 4。

### 4.3 为什么是 [2, 4]

| 上下界 | 理由 |
| --- | --- |
| 下界 2 | 1 个就没必要 fork（直接执行更快），子 Agent 至少要 2 个才有"并行价值" |
| 上界 4 | ① ThreadPoolExecutor 有 3 个 worker，超 3 排队 ② LLM API 并发限流通常 5 RPS 以下 ③ 前端展示 4 个 task_running 已经接近 UI 极限 |

**用户改源码绕过怎么办？**

绕过 `_clamp_subagent_limit` 是用户主动行为，但还有 Layer 3 兜底：即使 `max_concurrent=20`，ThreadPoolExecutor 还是只有 3 个 worker，超出排队。

## 5、Layer 3：ThreadPoolExecutor 物理上限

### 5.1 全局调度池

```python
# subagents/executor.py:151
_scheduler_pool = ThreadPoolExecutor(max_workers=3, thread_name_prefix="subagent-scheduler-")

MAX_CONCURRENT_SUBAGENTS = 3  # 这个常量被 SubagentLimitMiddleware 默认值引用
```

### 5.2 排队行为

如果 LLM 输出 8 个 task（中间件失效）：
- task 1-3：被 `_scheduler_pool.submit` 立即接受 → 在 3 个 worker 上跑
- task 4-8：进入 `_scheduler_pool` 内部队列 → 等 worker 空闲

**主 Agent 视角：** 8 个 `task_tool` 协程都在轮询，但只有 3 个对应的子 Agent 在真正跑。其他 5 个的 `result.status == PENDING` —— 状态卡在 PENDING 直到 worker 空闲。

### 5.3 daemon loop 共享

注意：`max_workers=3` 是 **scheduler 线程** 的上限，不是 **协程** 的上限。

```
3 个 scheduler 线程
    ├─ Thread 1: 调用 asyncio.run_coroutine_threadsafe → daemon loop
    ├─ Thread 2: 调用 asyncio.run_coroutine_threadsafe → daemon loop
    └─ Thread 3: 调用 asyncio.run_coroutine_threadsafe → daemon loop
    
daemon loop（一个）
    ├─ Coroutine 1 (Thread 1 提交)
    ├─ Coroutine 2 (Thread 2 提交)
    └─ Coroutine 3 (Thread 3 提交)
```

3 个协程在 daemon loop 里**并发**（不是并行）跑，靠 LLM 等待时让出控制权。

### 5.4 Future.result 阻塞

```python
def run_task():
    ...
    execution_future = _submit_to_isolated_loop_in_context(...)
    try:
        execution_future.result(timeout=self.config.timeout_seconds)  # ← 阻塞
    except FuturesTimeoutError:
        ...
```

每个 scheduler 线程**阻塞**等待自己的协程完成。所以 3 个 worker 在跑时整个调度池就 100% 占用。

## 6、三层防护的"AND" 逻辑

### 6.1 真值表

| Layer 1 | Layer 2 | Layer 3 | 实际并发上限 |
| --- | --- | --- | --- |
| ✅ Prompt 生效（LLM 输出 ≤3 task） | — | — | ≤ 3 |
| ❌ Prompt 失效（LLM 输出 8 task） | ✅ Middleware 截断到 3 | — | 3 |
| ❌ Prompt 失效 | ❌ Middleware 失效 | ✅ ThreadPool=3 | 3（其他排队） |
| ❌ 全部失效（用户改源码） | ❌ | ❌（max_workers=10） | 10（达到 LLM API 限流） |

**第 4 行说明：** 用户主动改源码到 `max_workers=10` 是"开发者自己负责"的事，DeerFlow 不再防。但 LLM API 还有自己的限流，会在更高一层兜底。

### 6.2 防护层的"独立失败"假设

设计原则：**每一层假设其他层失败**。

- Prompt 假设 Middleware 失败 → 自己给 LLM 一个明确的限制
- Middleware 假设 Prompt 失败 → 不依赖 LLM 配合，硬截断
- ThreadPool 假设两者都失败 → 物理上限制并发线程数

这是**纵深防御 (defense in depth)** 思想，借鉴自安全工程。

## 7、本章 ❓→💡 问答

### Q1：如果 LLM 输出 1 个 task 调用，触发什么？

**A：** 三层都不触发：
- Layer 1：prompt 说 ≤ 3，1 ≤ 3 OK
- Layer 2：`task_indices = [0]`，`1 ≤ self.max_concurrent (3)`，return None 不截断
- Layer 3：`_scheduler_pool.submit(run_task)` 立即接受，1 个 worker 跑

完全正常路径。

### Q2：如果 LLM 一轮输出 3 个 task，下一轮再输出 3 个 task，最多并发几个？

**A：** 上限取决于第一轮是否完成：

- **场景 A：** 第一轮 3 个还没完成，LLM 又被调用 —— 不会发生。LangGraph 是同步执行（一轮 LLM → 一轮 tool → 一轮 LLM），第二轮 LLM 调用前必须等第一轮的 3 个 task 都完成（ToolMessage 都到齐）。
- **场景 B：** 第一轮 3 个完成 → 第二轮再发 3 个。这是合法的"多批次执行" —— 总累计 6 个，但**任何时刻并发** ≤ 3。

ThreadPool 的 max_workers 是"瞬时并发"上限，不是"总累计"。

### Q3：`SubagentLimitMiddleware` 在 `subagent_enabled=False` 时不挂载，安全吗？

**A：** 安全。`subagent_enabled=False` 时主 Agent 拿不到 `task` 工具，LLM 看不到 `task` 在工具列表里，**不会输出 `task` tool_call**。

理论上可以：
- LLM 不看工具列表，幻觉式输出 `task` tool_call → ToolNode 找不到对应工具 → 抛 ToolException → ToolErrorHandlingMiddleware 捕获转 ToolMessage

整个链路不会真的启动子 Agent，所以 SubagentLimit 不挂载也安全。

### Q4：用户在 config 里写 `max_concurrent_subagents=1`，`_clamp` 钳制到 2，这是"违反用户意图"吗？

**A：** 是。但这个钳制的设计哲学是：**1 个不算"并发"，应该直接执行不要 fork**。

用户写 1 通常出于：
- 误以为"控制成本"（其实 1 个 task 比直接执行更耗 token）
- 害怕并发出错

DeerFlow 的态度：**"如果你只想要 1 个，请不要启用 subagent_enabled"**。强制下限 2 是引导用户走对路径。

### Q5：能不能根据下游 LLM 限流动态调整 max_concurrent？

**A：** 当前不支持。`max_concurrent` 是启动期固定的。

理论上可以加自适应：
- 监控最近 N 次 LLM 请求的 RPS / TPM
- 接近限流时 dynamic 降到 2
- 限流恢复升到 3

但 DeerFlow 选了简单：**固定 3 个 worker**，让 LLM provider 的限流接管（429 → LLMErrorHandlingMiddleware 重试）。

## 8、本章总结

**三层防护对比：**

| 层 | 类型 | 作用位置 | 失效后果 |
| --- | --- | --- | --- |
| Prompt 软约束 | 沟通层 | LLM 输出阶段 | LLM 多输出几个 task → Layer 2 兜底 |
| Middleware 硬截断 | 框架层 | after_model 钩子 | task 全部进入 ToolNode → Layer 3 兜底 |
| ThreadPool 物理上限 | 资源层 | execute_async submit | task 排队等 worker → 用户感知到延迟 |

**关键代码片段（背下来）：**

```python
# 软约束：System Prompt
"⛔ 硬性并发限制：每次响应最多 3 个 task 调用，这不是可选项。"

# 硬截断
def _truncate_task_calls(self, state):
    task_indices = [i for i, tc in enumerate(tool_calls) if tc.get("name") == "task"]
    if len(task_indices) <= self.max_concurrent:
        return None
    indices_to_drop = set(task_indices[self.max_concurrent:])
    truncated = [tc for i, tc in enumerate(tool_calls) if i not in indices_to_drop]
    return {"messages": [clone_ai_message_with_tool_calls(last_msg, truncated)]}

# 物理上限
_scheduler_pool = ThreadPoolExecutor(max_workers=3, ...)
MAX_CONCURRENT_SUBAGENTS = 3
```

> Prompt 软约束求自觉，Middleware 截断不商量，ThreadPool 物理三 worker —— 三层兜底，不怕模型瞎搞。
