# 06 19 中间件链 — Lead Agent 的拼装顺序与拦截点

> 面试口径：DeerFlow 的 Lead Agent 在 LangGraph `create_agent` 之上挂了 **19 个中间件**，分布在 4 个拦截点 (`before_agent` / `before_model` / `wrap_model_call` & `after_model` / `wrap_tool_call`)。中间件之间**互相依赖**，顺序错了就会出怪 bug —— 比如 SubagentLimit 必须在 LoopDetection **之前**（先截断多余的 task 再判循环），Clarification 必须**永远最后**（要拦截到最终的 tool_calls 列表）。这章把每个中间件的拦截点 + 排序原因 + 与子 Agent 通信的关联讲透。

**本章课程目标：**

- 理解 LangChain 中间件的 4 个拦截点 (`before_agent` / `before_model` / `wrap_model_call` / `wrap_tool_call`) 各自的语义
- 把 19 个中间件按"启动期 / 模型期 / 工具期"分类
- 重点解析与主子通信相关的 5 个：SubagentLimit / TokenUsage / LoopDetection / Clarification / Summarization
- 掌握"为什么这个顺序" 的因果链

**学习建议：** 这章不需要全背 19 个，但**与子 Agent 通信相关的 5 个必须熟**（重点章节）。其他 14 个知道职责即可，面试官不大会追问 ThreadDataMiddleware 怎么实现的。

### 2.2 关键排序原因

#### 排序 ① TokenUsage 在 SubagentLimit 之前

```python
# 顺序：6. TokenUsage → 11. SubagentLimit
```

**为什么？** TokenUsage 在 `after_model` **正向遍历**消息（找 ToolMessage 反向找 AIMessage 合并 token）；SubagentLimit 在 `after_model` 修改最后一条 AIMessage 的 tool_calls 列表。

**LangChain 的中间件执行顺序：** `after_model` 是**反向调用**（LIFO）—— 最后注册的最先执行。所以执行顺序实际是：

```
LLM 输出 AIMessage(tool_calls=[t1...t8])
        ↓
SafetyFinishReason after_model（如果有 unsafe finish_reason 清空 tool_calls）
        ↓
LoopDetection after_model（检测循环）
        ↓
SubagentLimit after_model（截断 t4-t8）→ AIMessage(tool_calls=[t1, t2, t3])
        ↓
DeferredToolFilter (wrap_model_call 阶段)
        ↓
TokenUsage after_model（记录 token，可能合并 t1-t3 子 Agent 的 usage）
        ↓
state.messages 更新
```

**为什么 TokenUsage 注册早但执行晚？**
- LangChain 的 `after_model` 钩子按"反向注册顺序"调用 —— 设计哲学是"先注册的中间件最贴近 LLM"
- TokenUsage 要看**最终的、清理过的 tool_calls 列表**，所以应该最贴近主流程（执行最晚）

#### 排序 ② SubagentLimit 在 LoopDetection 之前注册（but 执行更晚）

```python
# 注册顺序：11. SubagentLimit → 12. LoopDetection
# 实际执行：LoopDetection 先 → SubagentLimit 后
```

**为什么 SubagentLimit 要后执行？**

考虑场景：LLM 反复输出相同的 5 个 task 调用（陷入循环）。

- 如果 SubagentLimit 先执行 → 截断成 3 个 → LoopDetection 看到 3 个 task 调用，可能没识别为循环
- 如果 LoopDetection 先执行 → 看到原始 5 个 task → 正确识别循环 → SubagentLimit 再截断也无所谓

LoopDetection 应该看**模型原始意图**，不要被 SubagentLimit 干扰。

#### 排序 ③ ClarificationMiddleware 必须最后

```python
# 永远在最后注册
middlewares.append(ClarificationMiddleware())
```

**为什么？** Clarification 是 `wrap_tool_call` 拦截点，要拦截到**最终的、所有过滤后的** tool_calls。如果它在前面，后续 SubagentLimit / SafetyFinish 修改了 tool_calls，Clarification 看到的是旧版。

最后注册保证：所有 `after_model` 修改完成后，wrap_tool_call 看到最新的 tool_calls。

## 3、与子 Agent 通信直接相关的 5 个中间件

### 3.1 SubagentLimitMiddleware（重要）

```python
# subagent_limit_middleware.py
class SubagentLimitMiddleware(AgentMiddleware[AgentState]):
    """截断单次模型响应中超出上限的 task 工具调用。"""
    
    def __init__(self, max_concurrent: int = 3):
        super().__init__()
        self.max_concurrent = _clamp_subagent_limit(max_concurrent)  # [2, 4]
    
    def _truncate_task_calls(self, state: AgentState) -> dict | None:
        messages = state.get("messages", [])
        if not messages:
            return None
        
        last_msg = messages[-1]
        if getattr(last_msg, "type", None) != "ai":
            return None
        
        tool_calls = getattr(last_msg, "tool_calls", None)
        if not tool_calls:
            return None
        
        # 只统计 task 调用（其他工具不限）
        task_indices = [i for i, tc in enumerate(tool_calls) if tc.get("name") == "task"]
        if len(task_indices) <= self.max_concurrent:
            return None
        
        # 保留前 max_concurrent 个 task + 所有非 task 工具
        indices_to_drop = set(task_indices[self.max_concurrent:])
        truncated = [tc for i, tc in enumerate(tool_calls) if i not in indices_to_drop]
        
        logger.warning(f"Truncated {len(indices_to_drop)} excess task tool call(s)...")
        
        # clone AIMessage 替换原消息
        updated_msg = clone_ai_message_with_tool_calls(last_msg, truncated)
        return {"messages": [updated_msg]}
    
    @override
    def after_model(self, state, runtime):
        return self._truncate_task_calls(state)
    
    @override
    async def aafter_model(self, state, runtime):
        return self._truncate_task_calls(state)
```

**关键设计：**

| 设计点 | 说明 |
| --- | --- |
| `max_concurrent` 范围 [2, 4] | 钳制到合理区间 —— 太小没必要 fork，太大并发开销大 |
| 只统计 task 调用 | 其他工具（read_file / web_search）不参与计数，可以在同一响应里有 1 个 task + 5 个 read_file |
| 保留前 N 个 | 不是随机选，是**保留 LLM 最先想到的**（通常更重要） |
| 用 `clone_ai_message_with_tool_calls` | 保留原 AIMessage 的 id（让 LangGraph 识别为"替换"而不是"新增"） |

**面试可能追问：**

> Q: 为什么 max_concurrent 钳制到 [2, 4]？
> A: ① 下限 2：1 个就没必要 fork（直接执行更快）。② 上限 4：超过 4 个并发 LLM 调用容易触发供应商限流；ThreadPoolExecutor 也只有 3 个 worker，超过 3 个会排队 —— 4 是兼顾"配置弹性"和"实际能力"的折中。

### 3.2 TokenUsageMiddleware（重要）

```python
class TokenUsageMiddleware(AgentMiddleware):
    def _apply(self, state: AgentState):
        messages = state.get("messages", [])
        if not messages:
            return None
        
        # 反向遍历找连续的 ToolMessage
        state_updates: dict[int, AIMessage] = {}
        if len(messages) >= 2:
            from deerflow.tools.builtins.task_tool import pop_cached_subagent_usage
            
            idx = len(messages) - 2
            while idx >= 0:
                tool_msg = messages[idx]
                if not isinstance(tool_msg, ToolMessage) or not tool_msg.tool_call_id:
                    break  # 找到非 ToolMessage 就停
                
                subagent_usage = pop_cached_subagent_usage(tool_msg.tool_call_id)
                if subagent_usage:
                    # 反向找 AIMessage（带 tool_call_id 的）
                    dispatch_idx = idx - 1
                    while dispatch_idx >= 0:
                        candidate = messages[dispatch_idx]
                        if isinstance(candidate, AIMessage) and _has_tool_call(candidate, tool_msg.tool_call_id):
                            existing_update = state_updates.get(dispatch_idx)
                            prev = existing_update.usage_metadata if existing_update else (candidate.usage_metadata or {})
                            merged = {
                                **prev,
                                "input_tokens": prev.get("input_tokens", 0) + subagent_usage["input_tokens"],
                                "output_tokens": prev.get("output_tokens", 0) + subagent_usage["output_tokens"],
                                "total_tokens": prev.get("total_tokens", 0) + subagent_usage["total_tokens"],
                            }
                            state_updates[dispatch_idx] = candidate.model_copy(update={"usage_metadata": merged})
                            break
                        dispatch_idx -= 1
                idx -= 1
        ...
```

**关键设计：**

| 设计点 | 说明 |
| --- | --- |
| 反向遍历 ToolMessage | 一次 LLM 响应可能调多个工具，多个 ToolMessage 紧跟在一起 |
| 用 `tool_call_id` 索引 | 唯一标识，跨多个并发 task 调用都能正确归位 |
| `state_updates` dict 累加 | 5 个并发 task 都映射到同一个 AIMessage，结果是 5 个 token 累加 |
| `model_copy(update=...)` | Pydantic v2 风格，不修改原对象 |
| `pop_cached_subagent_usage` | 取出后立即移除，每个 tool_call_id 只 pop 一次 |

**为什么用 `pop` 不用 `get`？**
- 防止内存泄漏 —— 否则 `_subagent_usage_cache` 永远在涨
- 防止重复合并 —— 如果 state 被回放，第二次 get 会重复加 token

### 3.3 LoopDetectionMiddleware

`after_model` 拦截点：检测重复 tool_call 模式。

**为什么对子 Agent 通信重要？**

LLM 可能因为子 Agent 反复返回相同结果而陷入循环：
- 第 1 轮：5 个 task 调用 → 5 个相同的 ToolMessage 回流
- 第 2 轮：LLM 看到 5 个相同结果，再发 5 个 task 调用 → 又 5 个相同结果
- ...

LoopDetection 通过比对最近 N 轮的 tool_calls 哈希，发现重复后**强制中断**，避免无限循环。

### 3.4 ClarificationMiddleware

`wrap_tool_call` 拦截点：拦截 `ask_clarification` 调用，**中断 Agent 执行**让用户回答。

**为什么对子 Agent 通信重要？**

子 Agent 配置默认 `disallowed_tools=["task", "ask_clarification", "present_files"]`：

```python
# subagents/builtins/general_purpose.py:48
disallowed_tools=["task", "ask_clarification", "present_files"],
```

**禁止子 Agent 调用 `ask_clarification`** —— 因为：
- 子 Agent 在后台跑，没有"用户"对话方
- 即使能问，前端也不知道这是哪个子 Agent 在问
- 中断流会破坏 task_tool 的轮询逻辑

所以 ClarificationMiddleware 只在主 Agent 上工作；子 Agent 根本没有这个工具。

### 3.5 DeerFlowSummarizationMiddleware

`before_model` 拦截点：上下文超长时压缩历史消息。

**与子 Agent 的关系：**
- 主 Agent 的消息历史会因 ToolMessage（子 Agent 结果）膨胀
- 一个子 Agent 跑 10 分钟可能返回 5000 token 的结果
- 主 Agent 跑 5 个子 Agent 后消息历史可能 50k+ token
- Summarization 把旧 ToolMessage 总结成摘要，保留最近 N 轮原始消息

**子 Agent 不启用 Summarization** —— 因为子 Agent 任务通常单轮 / 短任务，不会爆上下文。子 Agent 的中间件链由 `build_subagent_runtime_middlewares` 构建，**不包含 Summarization / TokenUsage / SubagentLimit / Clarification 等主 Agent 专属中间件**。

## 5、本章 ❓→💡 问答

### Q1：为什么 TokenUsageMiddleware 不在子 Agent 上启用？

**A：** 子 Agent 的 token 用量由 `SubagentTokenCollector`（LangChain 回调）独立收集 → 写入 `_subagent_usage_cache` → 主 Agent 的 TokenUsageMiddleware 合并。

如果子 Agent 也启 TokenUsageMiddleware：
- 子 Agent 消息历史里没有 ToolMessage（它的工具是普通工具不是 task），合并逻辑无的放矢
- 重复记录到 RunJournal —— SubagentTokenCollector 已经记录了

所以子 Agent 用 collector 就够，不挂 TokenUsageMiddleware。

### Q2：SubagentLimit 截断的 tool_calls 会被框架认为"丢失"吗？

**A：** 不会。LangChain 的 ToolNode 看到 AIMessage 里有 N 个 tool_calls 就执行 N 个，被截断的 tool_calls 不在列表里 → 不执行 → 没有对应的 ToolMessage → LLM 下一轮不知道它们存在。

但 LLM 会**奇怪"我不是发了 5 个 task 吗，怎么只看到 3 个结果"**，可能重发。System prompt 里专门写了引导：

> "**多批次执行**（>3 个子任务时）：第 1 轮：并行启动子任务 1-3 → 等待结果。第 2 轮：并行启动下一批 → 等待结果。"

让 LLM 知道"分批是设计内的"。

### Q3：自定义中间件应该插在哪里？

**A：** `_build_middlewares` 接受 `custom_middlewares` 参数，会**插在 SafetyFinishReason 之前、Clarification 之前**：

```python
if custom_middlewares:
    middlewares.extend(custom_middlewares)

# Safety 之后
if safety_config.enabled:
    middlewares.append(SafetyFinishReasonMiddleware.from_config(safety_config))

# Clarification 永远最后
middlewares.append(ClarificationMiddleware())
```

这是合理位置：
- 在主流程中间件之后（业务逻辑做完）
- 在 Safety / Clarification 之前（不破坏安全和澄清逻辑）

### Q4：中间件链断了某一个会怎样？

**A：** 视中间件而定：
- `Token / Title / Memory` 缺失 → 数据丢失，不影响功能
- `SubagentLimit` 缺失 → LLM 一次响应 8 个 task 调用全部启动 → 线程池排队 / 超时
- `Clarification` 缺失 → ask_clarification 工具调用走默认逻辑（不中断 Agent，返回错误）—— 用户看不到澄清提示

`_build_middlewares` 里有大量 `if config.enabled:` 检查 —— 中间件可选启用，缺失不报错只丢功能。

## 6、本章总结

**记忆 19 中间件的两个抓手：**

1. **按拦截点分类**（4 类）：
   - `before_agent` (3 个)：ThreadData / Uploads / DynamicContext
   - `before_model` (4 个)：DanglingToolCall / Summarization / Todo / ViewImage
   - `wrap_model_call / after_model` (8 个)：LLMError / TokenUsage / SubagentLimit / LoopDetection / Title / SafetyFinish / DeferredToolFilter
   - `wrap_tool_call` (4 个)：Sandbox / Guardrail / ToolError / Clarification + SandboxAudit

2. **按子 Agent 通信关系分类**（5 个核心）：
   - SubagentLimit：截断超额 task
   - TokenUsage：合并子 Agent token 到 AIMessage.usage_metadata
   - LoopDetection：检测重复 task 循环
   - Clarification：拦截 ask_clarification 中断
   - Summarization：压缩 ToolMessage 历史

**核心顺序原则：**

```
TokenUsage 注册早 → 执行最晚（看最终 tool_calls）
SubagentLimit 注册晚 → 执行较晚（修改 tool_calls）
LoopDetection 注册更晚 → 执行更早（看原始意图）
```
