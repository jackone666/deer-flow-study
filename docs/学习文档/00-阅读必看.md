# 阅读必看 — DeerFlow 主子智能体通信机制学习指南

> **面试口径：** DeerFlow 是一个面向 AI Agent 应用的工程化运行时，核心是 **基于 LangGraph 的主子智能体编排系统**。系统实现主体是 Python 后端 (`backend/packages/harness/deerflow/`) + LangGraph Agent + 19 个中间件 + 沙箱 + SSE 流式网关。它不是一个 LangGraph 框架的简单封装，而是把"主智能体如何把任务派给子智能体、如何回收 token、如何取消、如何流式回传"这些**生产级问题**全部解决了的工程实现。

# 🔍 一、为什么需要这份资料？

学完 LangGraph 的 `create_agent` 之后，很多同学卡在同一个问题：**"然后呢？"**

跑通一个 Agent Demo 和做一个 **能扛真实流量、能调度 N 个子 Agent、能流式回传、能正确合并 token、能优雅取消** 的生产 Agent，中间隔着一堆系统化的坑：

- LangGraph 的 `create_agent` 只生成"一个图"，**多 Agent 协同的桥梁要自己建** — DeerFlow 怎么建的？
- LLM 一次响应吐出 8 个 `task` 调用怎么办？**`max_concurrent` 限流写在 prompt 里靠不靠谱？**
- 主 Agent 在 async 上下文里跑，子 Agent 想用同一个 httpx 连接池，**新建事件循环会把连接撕裂** — 怎么解？
- 子 Agent 烧的 token 怎么挂到调度它的那条 `AIMessage.usage_metadata` 上？**前端要按消息维度展示费用**。
- 用户点了"停止"，子 Agent 此刻正在跑一个 60 秒的工具调用，**`Future.cancel()` 是杀不死它的** — 怎么协作式退出？
- 子 Agent 默认不能再 fork 孙 Agent — 这个**递归防护是怎么写死的**？
- 父子智能体的沙箱、文件、上传图片怎么共享？**深拷贝 vs 引用传递**？

DeerFlow 的 `task_tool` + `SubagentExecutor` + `TokenUsageMiddleware` + `SubagentLimitMiddleware` 这一套就是回答这些问题的。这份资料把答案从源码里挖出来，直接让你读完就能讲。

# ✨ 二、核心价值与差异化

- **完整真实链路：** 不是"接个 task 工具跑 Demo"，而是从 LangGraph state → tool_call → 注册表 → Executor → ThreadPoolExecutor → 持久 daemon loop → astream → 协作取消 → token 上报 → SSE 推送 → ToolMessage 回传 → 中间件合并 usage_metadata 一条主线串到底。
- **大量架构图与代码片段：** 每个核心机制都配 Mermaid 图 + 真实源码片段（来自 `executor.py:474-668`、`task_tool.py:312-609`），帮你建立直觉而不是死记文字。
- **先讲 What，再讲 How，最后讲 Why：** 每个机制都按 "现象 → 实现 → 设计取舍" 三层展开。面试官问 "为什么不直接用 X 方案"，你能直接答出它的缺点。
- **对标真实面试：** 每章末尾都有 ❓→💡 问答块，覆盖 Agent 方向高频考点。第 10 章是 30 道整合题，按"基础 / 进阶 / 系统设计 / 故障排查"四档分级。

# 四、面试表达建议

学完后，建议把 DeerFlow 主子通信按下面四层来讲：

1. **架构定位：** DeerFlow 用 **"主智能体把子任务作为工具调用 (tool-call-as-delegation)"** 的范式实现父子通信。它不是 LangGraph 子图嵌套，也不是 Supervisor 模式，而是借用 LangChain 的 `@tool` 把 `task_tool` 注册成主 Agent 的一个普通工具，由 LLM 自己决定何时 fork。

2. **核心链路：** LLM tool_call → `task_tool` → `get_subagent_config` → `SubagentExecutor.execute_async` → ThreadPool → 持久 daemon loop → `_aexecute` → `agent.astream()` → 5s 轮询 → SSE 推送 → COMPLETED → `ToolMessage` 回传 → 下一轮 LLM。

3. **技术亮点：** 持久 daemon loop（避免 httpx 撕裂）、协作式取消（`threading.Event` + `asyncio.shield`）、Token 三层合并（缓存 + 全局 dict + 中间件回写）、三层并发防护（Prompt + Middleware + ThreadPool）、技能白名单父子取交集。

4. **边界诚实：** ① 子 Agent 默认不能 fork 孙 Agent（`subagent_enabled=False` 写死），如果要支持递归得改 `task_tool.py:434`。② 跨进程 / 跨机器扩展不在当前实现范围内，目前是单进程多线程。③ Token 合并依赖 `tool_call_id` 作为缓存键，子 Agent 异常崩溃时缓存可能残留（异常路径有 `_subagent_usage_cache.pop(tool_call_id, None)` 兜底）。

# 🎯 七、本资料的"硬骨头清单"

如果你只能记住 7 件事，记住这 7 件，面试基本不会被卡：

1. **`task_tool` 是父子通信的唯一桥梁**，没有别的通道（共享状态除外）。`runtime.state` 引用传递的 `sandbox` / `thread_data` 是隐式反向通信。
2. **子 Agent 在持久 daemon event loop 上跑**，不是每次新建 loop —— 因为 httpx / MCP 客户端的连接池绑定在 loop 上，loop 关闭会撕裂连接。
3. **协作式取消通过 `threading.Event` 在 `astream()` 迭代边界检查** —— `Future.cancel()` 杀不死正在跑工具调用的子线程，必须等子 Agent 自己退出。
4. **Token 三层合并：** `SubagentTokenCollector`（子 Agent 内）→ `_subagent_usage_cache`（全局 dict）→ `TokenUsageMiddleware.after_model`（合并到调度它的 `AIMessage.usage_metadata`）。索引键是 `tool_call_id`。
5. **并发三层防护：** Prompt 软约束（写在 `_build_subagent_section`）+ `SubagentLimitMiddleware` 硬截断（`after_model` 钩子里直接改 tool_calls 列表）+ `ThreadPoolExecutor(max_workers=3)`（物理上限）。
6. **递归防护写死在 `task_tool.py:434`：** `available_tools_kwargs["subagent_enabled"] = False` —— 子 Agent 拿不到 `task` 工具，所以不能再 fork。
7. **轮询 5 秒 + 超时 = 执行超时 + 60 秒缓冲：** 正常路径线程池超时先触发（`future.result(timeout=...)`），轮询超时是兜底防死锁。

读完这 10 章，再把这 7 条背下来，主子通信这一块面试就稳了。
