# 02 通信全景图 — 从用户输入到子 Agent 结果回传的完整 8 步

> 面试口径：DeerFlow 主子通信全程 8 步，可以分为 **三段**：① 主 Agent 决定 fork（LLM tool_call）② Executor 后台执行（持久 daemon loop）③ 结果回传（ToolMessage + token 合并）。掌握这张图，就能在面试里不看代码也能讲清楚 "用户问一句 → 子 Agent 跑完 → 主 Agent 给答案" 完整发生了什么。

**本章课程目标：**

- 把"用户输入 → 子 Agent 完成 → ToolMessage 回流 → LLM 继续 → 输出最终答案" 8 步串起来
- 知道每一步发生在哪个文件、哪个函数、哪一行
- 建立 "看到日志能反推处于第几步" 的能力（面试官常问的故障排查题就靠这个）

**学习建议：** 这张全景图建议**画 3 遍**：第 1 遍跟着本章读源码画一遍；第 2 遍合上本章自己画一遍；第 3 遍考别人讲（找同事 / 室友），讲不通的地方就是没吃透的环节。

## 1、本章导读

第 1 章建立了"工具调用即委派"的心智模型，但那只是**接口层**。真正决定 DeerFlow 行为的是**实现层** — 当 LLM 吐出一个 `task` tool_call 后，到底有多少个线程、多少个事件循环、多少次 lock 抢夺、多少次 SSE 推送、多少次 token 累加，这章把它们全部点亮。

## 3、八步详解

### Step 1：用户输入 → Worker → Lead Agent

**触发点：** 用户在前端发起一次对话。

**关键文件：** `runtime/runs/worker.py`

```python
async def run_agent(input_messages, config, ...):
    """Worker 是 LangGraph 图的执行驱动."""
    async for chunk in agent.astream(
        input={"messages": input_messages},
        config={"configurable": {...}, "callbacks": [run_journal]},
        stream_mode="values",
    ):
        # 把 chunk 通过 SSE 转发给前端
        yield serialize(chunk)
```

**关键点：**
- `agent` 是 `make_lead_agent(config)` 返回的 LangGraph 图
- `callbacks=[run_journal]` 注入 RunJournal（用于 token 统计 / 事件捕获）
- `stream_mode="values"` 让每次 state 更新都被 yield 出来，前端能看到中间过程

### Step 2：LLM 决定 fork → 输出 tool_call

**触发点：** Lead Agent 的 LLM 调用阶段，模型决定"我得拆解任务"。

**关键代码（System Prompt 片段，来自 `agents/lead_agent/prompt.py:281-429`）：**

```
**🚀 子代理模式已激活——拆解、委派、综合**
**⛔ 硬性并发限制：每次响应最多 3 个 task 调用，这不是可选项。**

# 用户问："对比 5 家云厂商"
# 思考：5 个子任务 → 需要多批次（每批最多 3 个）

# 第 1 轮：启动第一批 3 个
task(description="AWS 分析", prompt="...", subagent_type="general-purpose")
task(description="Azure 分析", prompt="...", subagent_type="general-purpose")
task(description="GCP 分析", prompt="...", subagent_type="general-purpose")
```

**关键点：**
- LLM 输出的是 `AIMessage(tool_calls=[{name: "task", args: {...}, id: "call_xxx"}])`
- 这条消息**先经过 `SubagentLimitMiddleware.after_model`** 检查是否超 3 个 task 调用，超了硬截断
- 然后被 LangGraph 框架自动路由到 `task_tool` 函数

### Step 3：task_tool 启动后台任务

**触发点：** LangGraph ToolNode 执行 `task` 工具。

**关键文件：** `tools/builtins/task_tool.py:312-458`

```python
@tool("task", parse_docstring=True)
async def task_tool(runtime, description, prompt, subagent_type, tool_call_id):
    # ── ① 解析配置 ──
    config = get_subagent_config(subagent_type, app_config=runtime_app_config)
    
    # ── ② 提取父 Agent 上下文 ──
    sandbox_state = runtime.state.get("sandbox")    # 引用传递
    thread_data = runtime.state.get("thread_data")  # 引用传递
    parent_model = metadata.get("model_name")
    trace_id = metadata.get("trace_id") or str(uuid.uuid4())[:8]
    
    # ── ③ 获取过滤后的工具列表 ──
    # 关键：subagent_enabled=False 防止递归
    tools = get_available_tools(model_name=effective_model, subagent_enabled=False, ...)
    
    # ── ④ 创建 Executor 并启动后台任务 ──
    executor = SubagentExecutor(config=config, tools=tools, ...)
    task_id = executor.execute_async(prompt, task_id=tool_call_id)
    
    # ── ⑤ 进入轮询循环（见 Step 6）──
    ...
```

**关键点：**
- `subagent_enabled=False` 是递归防护，**子 Agent 拿到的工具列表里没有 `task` 工具**
- 用 `tool_call_id` 直接当 `task_id` —— 一个 tool_call 对应一个子 Agent，天然唯一
- `sandbox` / `thread_data` 是**引用传递**，子 Agent 改了主 Agent 能看见

### Step 4：协程提交到持久 daemon loop

**触发点：** `executor.execute_async` 内部。

**关键文件：** `subagents/executor.py:753-815`

```python
def execute_async(self, task: str, task_id: str = None) -> str:
    # 1. 创建结果对象
    result = SubagentResult(task_id=task_id, status=SubagentStatus.PENDING)
    with _background_tasks_lock:
        _background_tasks[task_id] = result
    
    # 2. 提交到调度线程池
    parent_context = copy_context()  # 保留 ContextVar
    
    def run_task():
        # 3. 把协程提交到持久 daemon loop（不是新建 loop）
        execution_future = _submit_to_isolated_loop_in_context(
            parent_context,
            lambda: self._aexecute(task, result_holder),
        )
        try:
            execution_future.result(timeout=self.config.timeout_seconds)  # 阻塞等
        except FuturesTimeoutError:
            result_holder.cancel_event.set()  # 协作式取消
            result_holder.try_set_terminal(SubagentStatus.TIMED_OUT, ...)
            execution_future.cancel()
    
    _scheduler_pool.submit(run_task)  # 调度线程池（max_workers=3）
    return task_id
```

**关键点：**
- **两层线程**：调度线程池 (`_scheduler_pool`) → 持久 daemon loop (`_isolated_subagent_loop`)
- **持久 daemon loop**：全局只有一个，`run_forever()`，所有子 Agent 共享
- **为什么不新建 loop**：httpx / MCP 等异步客户端的连接池绑定在 loop 上，loop 关闭 = 连接撕裂
- **`timeout_seconds`**：默认 900 秒（15 分钟），超时后设置 cancel_event 协作式退出

### Step 5：子 Agent 执行（_aexecute 主循环）

**触发点：** 协程在持久 daemon loop 中被调度执行。

**关键文件：** `subagents/executor.py:474-668`

```python
async def _aexecute(self, task: str, result_holder: SubagentResult = None):
    # ① 构造初始 state（system_prompt + 任务 + 共享 sandbox/thread_data）
    state, filtered_tools = await self._build_initial_state(task)
    agent = self._create_agent(filtered_tools)  # 新的 LangGraph Agent
    
    # ② 注入 token collector（LangChain 回调）
    collector = SubagentTokenCollector(caller=f"subagent:{self.config.name}")
    run_config = {
        "recursion_limit": self.config.max_turns,
        "callbacks": [collector],
        "tags": [f"subagent:{self.config.name}"],
    }
    
    # ③ 流式执行子 Agent
    async for chunk in agent.astream(state, config=run_config, stream_mode="values"):
        # 协作式取消：每个 chunk 边界检查
        if result.cancel_event.is_set():
            result.try_set_terminal(SubagentStatus.CANCELLED, ...)
            return result
        
        final_state = chunk
        # 捕获 AIMessage 给 SSE 推送（去重防止重复发送）
        if isinstance(last_message, AIMessage) and not is_duplicate:
            ai_messages.append(last_message.model_dump())
    
    # ④ 提取最终结果
    final_result = self._extract_final_result(final_state)
    result.try_set_terminal(SubagentStatus.COMPLETED, result=final_result, token_usage_records=collector.snapshot_records())
```

**关键点：**
- 子 Agent 用 `create_agent` 创建，**和主 Agent 同一个工厂、同样的 state_schema (ThreadState)、同样的中间件构建函数 `build_subagent_runtime_middlewares`**
- `stream_mode="values"` —— 每次 state 更新都拿到完整 state（不是增量），方便取最后一条 AIMessage
- **协作式取消**：在 `astream` 的每个迭代边界检查 `cancel_event` —— 一个工具调用如果跑了 60 秒，cancel 不会立刻生效，要等下一次 chunk

### Step 6：主 Agent 轮询 + SSE 推送

**触发点：** Step 3 中 `task_tool` 启动 Executor 后进入的轮询循环。

**关键文件：** `tools/builtins/task_tool.py:475-575`

```python
writer = get_stream_writer()  # SSE 推送器
writer({"type": "task_started", "task_id": task_id, "description": description})

while True:
    result = get_background_task_result(task_id)
    
    # 实时推送子 Agent 的 AI 消息
    ai_messages = result.ai_messages or []
    if len(ai_messages) > last_message_count:
        for i in range(last_message_count, len(ai_messages)):
            writer({"type": "task_running", "task_id": task_id, "message": ai_messages[i], ...})
        last_message_count = len(ai_messages)
    
    # 检查终止状态
    usage = _summarize_usage(result.token_usage_records)
    if result.status == SubagentStatus.COMPLETED:
        _cache_subagent_usage(tool_call_id, usage, enabled=cache_token_usage)
        _report_subagent_usage(runtime, result)
        writer({"type": "task_completed", "task_id": task_id, "result": result.result, "usage": usage})
        cleanup_background_task(task_id)
        return f"Task Succeeded. Result: {result.result}"
    elif result.status == SubagentStatus.FAILED:
        ... return f"Task failed. Error: {result.error}"
    elif result.status == SubagentStatus.CANCELLED:
        ... return "Task cancelled by user."
    elif result.status == SubagentStatus.TIMED_OUT:
        ... return f"Task timed out. Error: {result.error}"
    
    await asyncio.sleep(5)  # 每 5 秒轮询一次
    poll_count += 1
    if poll_count > max_poll_count:  # 兜底超时
        request_cancel_background_task(task_id)
        ... return f"Task polling timed out after {timeout_minutes} minutes."
```

**关键点：**
- **轮询间隔 5 秒** —— 不是无脑死循环，而是 `await asyncio.sleep(5)`（让出控制权给其他协程）
- **SSE 推送 4 类事件**：`task_started` / `task_running` / `task_completed` / `task_failed` / `task_cancelled` / `task_timed_out`
- **去重推送**：`last_message_count` 跟踪已推送的消息数，只推增量
- **双层超时**：①`Future.result(timeout=)` 在 Step 4 触发 ②轮询计数超 `max_poll_count` 兜底（防止后台死锁）

### Step 7：ToolMessage 回流到主 Agent

**触发点：** `task_tool` return 字符串后，LangGraph 框架自动包装。

**框架行为（LangGraph ToolNode 内部）：**
```python
# task_tool 返回 "Task Succeeded. Result: ..." 字符串
# LangGraph 自动包装成：
ToolMessage(
    content="Task Succeeded. Result: ...",
    tool_call_id="call_xxx",
    name="task",

# 加到 state.messages 列表末尾
```

**关键点：**
- 主 Agent 的 LLM 在下一轮 `before_model` 看到这条 ToolMessage 就知道子 Agent 完事了
- ToolMessage 的 `tool_call_id` 和 AIMessage 中的 tool_call id 一一对应

### Step 8：Token 合并 + 最终回答

**触发点：** 主 Agent 下一轮 LLM 调用前后，`TokenUsageMiddleware` 的 `after_model` 钩子。

**关键文件：** `agents/middlewares/token_usage_middleware.py:275-323`

```python
class TokenUsageMiddleware(AgentMiddleware):
    def _apply(self, state: AgentState):
        messages = state.get("messages", [])
        
        # 反向遍历找连续的 ToolMessage
        idx = len(messages) - 2
        while idx >= 0:
            tool_msg = messages[idx]
            if not isinstance(tool_msg, ToolMessage):
                break
            
            # 从全局缓存中 pop 这个 tool_call_id 的 usage
            subagent_usage = pop_cached_subagent_usage(tool_msg.tool_call_id)
            if subagent_usage:
                # 反向找到调度这个 tool_call 的 AIMessage
                dispatch_idx = idx - 1
                while dispatch_idx >= 0:
                    candidate = messages[dispatch_idx]
                    if isinstance(candidate, AIMessage) and _has_tool_call(candidate, tool_msg.tool_call_id):
                        # 合并 usage_metadata
                        prev = candidate.usage_metadata or {}
                        merged = {
                            **prev,
                            "input_tokens": prev.get("input_tokens", 0) + subagent_usage["input_tokens"],
                            "output_tokens": prev.get("output_tokens", 0) + subagent_usage["output_tokens"],
                            "total_tokens": prev.get("total_tokens", 0) + subagent_usage["total_tokens"],
                        }
                        state_updates[dispatch_idx] = candidate.model_copy(update={"usage_metadata": merged})
                        break
                    dispatch_idx -= 1
            idx -= 1
```

**关键点：**
- **合并方向**：把子 Agent 的 token 加到 **调度它的那条 AIMessage** 的 `usage_metadata` 上
- **多个并发 task 调用**：5 个子 Agent 都挂到同一条 AIMessage（因为它们都是同一个 LLM 响应吐出来的）
- **前端展示**：每条 AIMessage 显示自己 + 所有子 Agent 累加的 token —— 这就是为什么按 tool_call_id 索引

## 5、本章 ❓→💡 问答

### Q1：为什么 SSE 推送和轮询用同一个循环？为什么不开两个协程？

**A：** 因为前端展示的"实时性"和"完成事件"必须**严格有序**。如果 SSE 推送在协程 A，轮询在协程 B：
- B 检测到 COMPLETED，return 给主 Agent
- A 的最后一波 task_running 还在 SSE 队列里没发完
- 前端收到 task_completed 后又收到 task_running —— UI 错乱

合并到一个 `while True` 里，每次循环先推 SSE 再检查状态，状态变 COMPLETED 时已经推完所有消息再 return，**保证前端永远先看到所有 task_running 再看到 task_completed**。

### Q2：5 秒轮询是不是太慢了？为什么不 1 秒？

**A：** 5 秒是工程权衡：
- **1 秒**：每个 task 在 15 分钟内会触发 900 次轮询，每次轮询要拿一次锁 (`_background_tasks_lock`)，CPU 浪费在没意义的检查上
- **10 秒**：用户从前端点"取消"到子 Agent 真正退出最坏要等 10 秒，体验差
- **5 秒**：折中 —— 每个 task 最多 180 次轮询，取消最多延迟 5 秒（实际上 cancel_event 检查是在每个 LLM chunk 边界，通常 < 1 秒）

而且子 Agent 的实时消息推送**不靠轮询**，是 `_aexecute` 内部每个 chunk 边界把 AIMessage 写入 `result.ai_messages`，5 秒轮询只是**拉取增量**。

### Q3：Step 6 里的 `_cache_subagent_usage` 和 `_report_subagent_usage` 是不是重复了？

**A：** 不重复，两个目的：
- **`_cache_subagent_usage`**：写入全局 `_subagent_usage_cache: dict[str, dict[str, int]]`，索引键是 `tool_call_id`，**给 Step 8 的 TokenUsageMiddleware 用** —— 让 token 出现在 AIMessage.usage_metadata 上
- **`_report_subagent_usage`**：写入 RunJournal（持久化记账本），按 caller 分桶 (`subagent:general-purpose` / `lead_agent` / `middleware:summarize`)，**用于事后审计 / 计费 / 监控**

一个面向消息（前端展示用），一个面向账单（后端审计用）。

## 6、本章总结

**8 步顺口溜：**

```
1. 用户问，Worker 启 → astream
2. LLM 想拆，输出 → tool_call(task)
3. task_tool 解配置 → 提共享 → 给 Executor
4. 协程提交 → daemon loop（持久不关）
5. 子图开跑 → 边跑边推 → cancel 边检
6. 主 Agent 5 秒轮 → SSE 推 → COMPLETED 返
7. 框架包 ToolMessage → 进 messages 队
8. TokenMiddleware 合 → AIMessage 携 usage
```
