## 2、生产监控指标（12 个必盯）

### 2.1 业务指标（4 个）

| 指标 | 含义 | DeerFlow 来源 | SLO 建议 |
| --- | --- | --- | --- |
| QPS | 每秒请求数 | Gateway 入口埋点 | 看容量规划 |
| 任务完成率 | RunStatus.success / 总数 | RunStore 聚合 | > 99% |
| 用户满意度 | (+1) / ((+1)+(-1)) | FeedbackRepository | > 80% |
| 平均响应时间 | end - start (含 SSE) | RunRecord.created_at vs updated_at | P95 < 60s |

### 2.2 LLM 指标（4 个）

| 指标 | 含义 | DeerFlow 来源 |
| --- | --- | --- |
| LLM 错误率 | LLM 调用失败次数 / 总调用 | LLMErrorHandlingMiddleware |
| LLM 平均延迟 | 单次 LLM 调用 P50/P95/P99 | RunJournal.llm_latency |
| Token 用量 | input/output/total 累计 | RunJournal._total_tokens |
| 模型分布 | 各模型调用占比 | RunJournal 按 model_name 分组 |

### 2.3 Agent 行为指标（4 个）

| 指标 | 含义 | DeerFlow 来源 |
| --- | --- | --- |
| 子 Agent fork 率 | 启用 task 的 run / 总 run | TokenUsage subagent_tokens > 0 计数 |
| 平均 Turn 数 | LLM 调用次数 / Run | RunJournal._llm_call_count |
| 工具错误率 | ToolMessage(status=error) / 总工具调用 | RunJournal tool_end 事件 |
| 超时率 | RunStatus.timeout / 总数 | RunStore 聚合 |

## 3、LLM 抖动降级（DeerFlow 4 种实现）

### 3.1 LLM 故障类型

LLM 服务可能遇到：

| 故障 | 现象 | 严重度 |
| --- | --- | --- |
| 限流（429） | 短时间内被限速 | 中（重试可恢复） |
| 服务超时 | 30 秒不返回 | 中 |
| 内容过滤 | finish_reason=content_filter | 高（无法靠重试） |
| 模型 OOM | 5xx 错误 | 高 |
| 连接错误 | 网络抖动 | 低（重试可恢复） |
| 截断（length） | finish_reason=length | 中 |

### 3.2 DeerFlow 已实现：LLMErrorHandlingMiddleware

```python
# agents/middlewares/llm_error_handling_middleware.py（关键片段）
class LLMErrorHandlingMiddleware(AgentMiddleware):
    # 默认配置
    retry_max_attempts: int = 3
    retry_base_delay_ms: int = 1000   # 第一次 1s
    retry_cap_delay_ms: int = 8000     # 最大 8s
    
    def _build_retry_delay_ms(self, attempt, exc):
        retry_after = _extract_retry_after_ms(exc)  # 从 429 响应头取 Retry-After
        if retry_after is not None:
            return retry_after
        backoff = self.retry_base_delay_ms * (2 ** max(0, attempt - 1))
        return min(backoff, self.retry_cap_delay_ms)
    
    async def awrap_model_call(self, request, handler):
        for attempt in range(1, self.retry_max_attempts + 1):
            try:
                return await handler(request)
            except (RateLimitError, APITimeoutError, ConnectionError) as e:
                if attempt >= self.retry_max_attempts:
                    # 已经重试到上限，转为 fallback ToolMessage
                    return self._fallback_message(e)
                delay = self._build_retry_delay_ms(attempt, e)
                await asyncio.sleep(delay / 1000)
```

**关键：**
- **指数退避**：1s → 2s → 4s → 8s（cap）
- **Retry-After 优先**：服务器返回 Retry-After 就听它的
- **超过上限不抛**：返回降级消息（让 Agent 继续，不让整个 run 失败）

### 3.3 DeerFlow 已实现：LoopDetectionMiddleware

```python
# 检测重复 tool_call 模式
class LoopDetectionMiddleware(AgentMiddleware):
    def after_model(self, state, runtime):
        # 取最近 N 轮的 AIMessage tool_calls
        recent_calls = self._extract_recent_tool_calls(state["messages"])
        # 比对哈希 / 编辑距离
        if self._is_repetitive(recent_calls):
            # 强制中断 + 注入提示
            return {"messages": [
                AIMessage(content="检测到工具调用循环，已中断。请重新组织任务。")
            ]}
```

**用途：** LLM 因子 Agent 反复返回相同结果而陷入死循环时，强制中断（见第 6 章 §3.3）。

### 3.4 DeerFlow 已实现：SafetyFinishReasonMiddleware

```python
class SafetyFinishReasonMiddleware(AgentMiddleware):
    def after_model(self, state, runtime):
        last = state["messages"][-1]
        if not isinstance(last, AIMessage):
            return None
        
        finish_reason = last.response_metadata.get("finish_reason")
        if finish_reason in self.unsafe_finish_reasons:  # ["content_filter", "safety", ...]
            # 清空 tool_calls（防止越界执行）
            return {"messages": [
                last.model_copy(update={"tool_calls": []})
            ]}
```

**用途：** 当 LLM 因为安全过滤而终止（finish_reason=content_filter）时，**清空 tool_calls 防止半执行**。

### 3.5 DeerFlow 已实现：ToolErrorHandlingMiddleware

```python
class ToolErrorHandlingMiddleware(AgentMiddleware):
    async def awrap_tool_call(self, request, handler):
        try:
            return await handler(request)
        except ToolException as e:
            # 把异常转成 ToolMessage（让 LLM 看到错误，自己调整）
            return ToolMessage(
                content=f"Tool failed: {e}",
                tool_call_id=request.tool_call_id,
                status="error",
            )
```

**用途：** 工具异常不抛崩整个 run，转成 ToolMessage 让 LLM 知道并 retry。

### 3.6 4 种降级策略汇总

```
故障类型           ↓ DeerFlow 处理
───────────────────────────────────────
限流 / 超时 / 网络 → LLMErrorHandling 重试 + 退避 + Retry-After
重复循环           → LoopDetection 检测 + 中断
安全过滤           → SafetyFinishReason 清空 tool_calls
工具异常           → ToolErrorHandling 转 ToolMessage
```

### 3.7 缺失的降级（需要补）

DeerFlow 当前**没有**：

| 缺失功能 | 怎么补 |
| --- | --- |
| 模型级降级（GPT-4 → GPT-4-mini → GPT-3.5） | LLMError 中间件加 fallback chain |
| 多 provider 切换（OpenAI 挂了切 Anthropic） | 改 `create_chat_model` 加多源支持 |
| 限流软上限（用户级 RPS 控制） | 加 ratelimit middleware（如 SlowAPI） |
| 突发流量熔断（Hystrix 风格） | 加 circuit breaker pattern |

**面试时可以提：** "DeerFlow 当前是单 provider + 简单重试。生产环境我会再加 model fallback chain 和 user-level rate limit。"

## 4、Token 成本控制（5 招）

### 4.1 招 1：Summarization 中间件压缩历史

DeerFlow 已实现：

```python
# agents/middlewares/summarization_middleware.py
DeerFlowSummarizationMiddleware(
    model="gpt-4o-mini",       # 用便宜模型摘要
    trigger=("token_count", 100_000),  # 100k token 触发
    keep=("messages", 20),     # 保留最近 20 条
)
```

**效果：** 长对话的 token 不会无限增长，保持在 100k 以下。

**成本节约：** 100k input * $5/1M = $0.50/次 × 多轮 → 节约 80%+

### 4.2 招 2：Token Budget 中间件（DeerFlow 没实现，可补）

```python
# 提议的 deerflow/agents/middlewares/token_budget_middleware.py
class TokenBudgetMiddleware(AgentMiddleware):
    def __init__(self, max_tokens_per_run: int = 1_000_000):
        self.max = max_tokens_per_run
    
    def after_model(self, state, runtime):
        usage = state.get("total_tokens_used", 0)
        last_usage = state["messages"][-1].usage_metadata or {}
        usage += last_usage.get("total_tokens", 0)
        
        if usage > self.max:
            # 强制结束
            return {"messages": [
                AIMessage(content=f"已达到 token 预算上限 {self.max}，本次 Run 结束。")
            ]}
        return {"total_tokens_used": usage}
```

**用途：** 防止单个 Run 因为陷入循环 / 复杂任务消耗过量 token（成本失控）。

### 4.3 招 3：模型路由（按任务类型选模型）

```python
# 简单任务 → 便宜模型
# 复杂任务 → 强模型

if task_type == "summarize" or task_type == "format":
    model = "gpt-4o-mini"     # $0.15/1M input
```

**可以补的：** 加 `RoutingMiddleware`，根据用户问题特征（关键词 / 长度 / embedding）选模型。

### 4.4 招 4：Prompt Cache（DeerFlow 利用了）

Anthropic / OpenAI 都支持 prompt cache：相同前缀的 system prompt 缓存，第二次调用 input token 折扣 90%。

**DeerFlow 的设计：**
- system prompt 静态化（不用变量插值）
- 用 `DynamicContextMiddleware` 把日期 / 记忆塞进**第一条 HumanMessage** 而不是 system prompt
- 这样 system prompt 永不变 → cache 命中率高

```python
# agents/lead_agent/agent.py 注释
# Always inject current date (and optionally memory) as <system-reminder> into the
# first HumanMessage to keep the system prompt fully static for prefix-cache reuse.
```

**效果：** 长 system prompt（10k token）只算第一次费用，后续 Run 都命中 cache。

### 4.5 招 5：DeferredTool 隐藏 schema

见第 15 章 §5.4。50 个 MCP 工具默认隐藏 schema，节约 10k+ token / 调用。

### 4.6 5 招汇总

```
1. Summarization：长对话压缩          → 节约 80% history tokens
2. TokenBudget：强制上限              → 防止失控
3. ModelRouting：任务级选模型          → 节约 70% 模型成本（简单任务用 mini）
4. PromptCache：system prompt 静态化   → 节约 90% input tokens（命中 cache）
5. DeferredTool：MCP schema 按需暴露   → 节约 10k+ token / 调用
```

```

  - DeferredTool：移除 15k MCP schema
  - Summarization：保持上下文紧凑
```

## 5、A/B 测试与灰度

### 5.1 Agent A/B 测试场景

**典型问题：** 改了 system prompt / 加了新 skill / 换了模型 —— 怎么知道**真的更好**？

不能只靠"自己测试觉得好"，要量化对比。

### 5.2 三种 A/B 设计

#### 方案 A：用户级分桶（最常用）

```python
def select_agent_version(user_id: str) -> str:
    # 按 user_id hash 分桶
    bucket = hash(user_id) % 100
    if bucket < 10:
        return "v_next"    # 10% 流量到新版本
    return "v_current"      # 90% 留旧版本
```

**优点：** 同一用户始终在同一桶（用户体验一致）。
**缺点：** 用户行为差异可能让分桶失衡。

#### 方案 B：请求级随机

```python
def select_agent_version() -> str:
    if random.random() < 0.1:
        return "v_next"
    return "v_current"
```

**优点：** 流量绝对随机分布。
**缺点：** 同用户两次请求可能用不同版本，体验不一致。

#### 方案 C：Shadow 模式

```python
async def run_request(input):
    # 主路径：v_current 跑，结果给用户
    result_current = await run_agent("v_current", input)
    
    # Shadow：v_next 也跑（结果不返回，只记录用于对比）
    asyncio.create_task(run_agent_shadow("v_next", input))
    
    return result_current
```

**优点：** 不影响用户体验，能拿到同输入下的对比数据。
**缺点：** 计算成本翻倍。

### 5.3 评估指标

A/B 期间监控：

| 指标 | 期望方向 |
| --- | --- |
| user_rating (+1) 比例 | 上升 |
| 平均 token 消耗 | 不显著上升 |
| 平均响应时间 | 不显著上升 |
| run 错误率 | 不显著上升 |

**统计显著性：** 用 t-test 或 Mann-Whitney U test，p < 0.05 才算赢。

### 5.4 DeerFlow 怎么实现 A/B（可补）

DeerFlow 当前**没有内置 A/B 框架**，可以加：

```python
# 假设新增 deerflow/experiments/ab_test.py
class ABTestMiddleware(AgentMiddleware):
    def __init__(self, experiments: list[Experiment]):
        self.experiments = experiments
    
    def before_agent(self, state, runtime):
        user_id = runtime.context.get("user_id")
        for exp in self.experiments:
            variant = exp.assign_variant(user_id)
            # 注入到 metadata，供后续中间件读取
            state.setdefault("ab_variants", {})[exp.name] = variant
```

**业务侧用法：**

```python
class TitleMiddleware:

        else:
            self.title_prompt = OLD_TITLE_PROMPT
```

## 6、故障排查 5 大案例

### 6.1 案例 1：Token 暴涨告警

**现象：** 单用户 1 小时 token 消耗 5M（正常 < 100k），账单警报。

**排查：**

1. **查 RunJournal**：定位是哪个 user_id / thread_id
2. **看 trace 详情**：通过 `event_store.get_events(thread_id)` 拿到完整 trace
3. **典型场景**：
   - LLM 陷入工具调用循环（每轮调 5 个相同的 task）
   - 子 Agent 内部又陷入循环（subagent_tokens 占大头）
   - Summarization 没触发（消息历史无限增长）

**根因定位：**

```sql
SELECT thread_id, total_tokens, llm_call_count, lead_agent_tokens, subagent_tokens
FROM runs
WHERE user_id = 'abnormal_user'
ORDER BY created_at DESC
LIMIT 10;
```

**应对：**
- 短期：手动 cancel 该 thread，加 user_id 到 rate limit 黑名单
- 中期：加 TokenBudget 中间件，单 run 上限 1M
- 长期：LoopDetection 阈值调严

### 6.2 案例 2：子 Agent 卡死不退出

**现象：** Run 状态停在 `running` 30 分钟，前端转圈。

**排查：**

1. **查 RunRecord**：`status=running`，`updated_at` 30 分钟前
2. **查后台任务**：`get_background_task_result(task_id)` → 状态是什么？
3. **典型场景**：
   - 子 Agent 卡在长 bash 调用（cancel_event 边界检查未触发）
   - daemon loop 内某个协程死锁
   - `request_cancel_background_task` 调了但子 Agent 在不可中断段

**应对：**
- 强制 timeout：把 `timeout_seconds` 调短（900 → 300）
- 给重型工具加内部 timeout（subprocess.run timeout=60）
- 主动 kill：操作员手动调 `request_cancel_background_task`

### 6.3 案例 3：SSE 连接频繁断开

**现象：** 用户反馈"刷新页面才能看到新对话，对话过程中没流式效果"。

**排查：**

1. **看 LB / Nginx 日志**：是否有 504 / 502
2. **查 StreamBridge 心跳**：是否每 15 秒发了 `__heartbeat__`
3. **查 Worker 状态**：是否还活着

**典型根因：**
- LB 默认 60 秒空闲超时，长任务 60 秒没事件就被切
- 客户端 Last-Event-ID 没正确传递，重连后丢事件

**应对：**

```python
# stream_bridge 心跳间隔调短
heartbeat_interval = 10  # 默认 15 → 10

# Nginx 配置
proxy_read_timeout 600;
proxy_send_timeout 600;
proxy_buffering off;  # ← 关键：禁用 buffering
```

### 6.4 案例 4：多 worker 状态混乱

**现象：** 用户连续两条消息，第二条找不到第一条的上下文（"你好"被忘了）。

**排查：**

1. **看 RunRecord**：两条 run 是否在同一 worker
2. **查 Checkpointer**：是 in-memory 还是 SQLite/Postgres

**典型根因：**
- 多 worker 部署 + 用 InMemorySaver checkpointer → worker 1 写的状态 worker 2 读不到
- 没配置 sticky session → 两条请求被路由到不同 worker

**应对：**
- Checkpointer 换 PostgresSaver（共享数据库）
- LB 配置 ip_hash 或 session sticky
- 参考第 13 章 §6.3

### 6.5 案例 5：LLM API 全面挂了

**现象：** OpenAI / Anthropic 全宕，所有 Run 失败。

**排查：**

1. **看 LLM 错误率**：突然 100%
2. **看官方 status page**：[status.openai.com](https://status.openai.com)

**应对：**
- **熔断器**：错误率 > 50% 时停止发请求 30 秒（避免雪崩）
- **降级模型**：OpenAI 挂了切 Claude / DeepSeek
- **降级回应**：直接返回"AI 服务暂时不可用，请稍后重试"

**DeerFlow 当前：** 没有 multi-provider fallback。可以这样补：

```python
# 假设的 deerflow/models/multi_provider.py
class FallbackChatModel:
    def __init__(self, primary: ChatModel, fallbacks: list[ChatModel]):
        self.primary = primary
        self.fallbacks = fallbacks
    
    async def ainvoke(self, *args, **kwargs):
        for model in [self.primary] + self.fallbacks:
            try:
                return await model.ainvoke(*args, **kwargs)
            except (APIError, TimeoutError):
                continue
        raise RuntimeError("All providers failed")
```

## 8、本章 ❓→💡 问答

### Q1：怎么衡量一个 Agent 项目的"健康度"？

**A：** 我会建一个 SLO 仪表盘，盯 4 类指标：

1. **业务**：任务完成率（>99%）、用户满意度（>80%）、P95 响应时间（<60s）
2. **质量**：LLM 错误率（<5%）、工具错误率（<2%）、Loop 触发率（<1%）
3. **成本**：单 Run 平均 token、token 同比、模型分布
4. **稳定**：超时率（<0.5%）、worker 健康度、Checkpointer 写延迟

**核心思路：业务指标 = 用户感知；其他三类 = 内部健康。** 业务跌了一定先修业务问题，其他指标只是辅助定位。

### Q2：如果一个用户狂刷 Agent，怎么防御？

**A：** 三层防护：

1. **基础设施层**：LB 加 IP rate limit（100 RPS / IP）
2. **应用层**：FastAPI middleware 加 user_id rate limit（10 RPS / user）
3. **业务层**：Token 预算上限（单用户每天 100k token / 单 run 1M token）

DeerFlow 当前**只有第 3 层的 token 监控**，没有 1/2 层 rate limit。生产部署需要在 Gateway 层加 SlowAPI 或类似中间件。

### Q3：Token 成本失控的根因通常是什么？

**A：** 排序：

1. **Loop（最常见）**：LLM 陷入循环，反复调用相同工具
2. **超长 prompt**：上传大文件全文塞进 prompt
3. **子 Agent 数量爆炸**：max_concurrent 没限制 + LLM 任性
4. **重复对话**：用户连发相似问题，没有结果缓存
5. **模型选错**：复杂任务用 Opus（贵），简单任务也用 Opus

**应对优先级：**
- LoopDetection 中间件（必备）
- TokenBudget 中间件（强烈推荐）
- 文件上传大小限制（前端 + 后端双重）
- SubagentLimit 中间件（DeerFlow 已实现）
- 结果缓存（按 query hash + thread_id 缓存）

### Q4：A/B 测试需要跑多久才有结论？

**A：** 看流量：

```
所需样本 ≈ 16 / (effect_size² × p × (1 - p))

例：
- effect_size = 0.05（5% 改善）
- p = 0.8（基线满意度）
- 需 16 / (0.0025 × 0.8 × 0.2) = 40,000 个样本

如果 1 万 DAU：4 天
如果 1 万 MAU：30 天
```

**实战经验：** 至少跑 7 天（覆盖周期性效应），最少 5000 个样本。

### Q5：故障复盘（postmortem）应该写什么？

**A：** 5 部分：

1. **Impact**：影响多少用户、持续多久、损失多少
2. **Timeline**：故障发生 → 检测 → 定位 → 修复 → 验证（带时间戳）
3. **Root Cause**：根因（不是表面现象）
4. **What Went Well**：什么环节做对了（不止讲坏的）
5. **Action Items**：每条都有 owner + deadline + 状态

**典型反模式：** "操作员失误"、"运气不好" —— 这些不是根因。要追问 **"为什么操作员能失误？" "为什么运气会影响系统？"**

## 9、本章总结

**生产实践 5 大支柱：**

```
1. 监控：12 指标 + Prometheus + Grafana
2. 降级：4 种中间件（LLMError / Loop / Safety / ToolError）

5. 故障：5 大案例排查思路
```

**面试金句：**

> "DeerFlow 的生产实践分两块：**已实现的稳定性中间件**（LLMError 重试退避 / LoopDetection 死循环检测 / SafetyFinish 不安全终止 / ToolError 转 ToolMessage）+ **生产部署该补的**（Prometheus 监控 / TokenBudget / 多 provider fallback / A/B 框架 / Sticky session 多 worker 部署）。
>
> 框架已经做了**LLM 抖动的自动恢复**，但**成本控制和可观测性需要部署侧补足**。这是开源框架的合理边界 —— 框架不应假设具体的监控栈和 SaaS。"
